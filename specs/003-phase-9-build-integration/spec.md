# Gravity DSL — Specification (Phase 9: Build integration — MSBuild target + reference sample emitter)

**Status:** Locked for implementation
**Date:** 2026-05-18
**Driven by:** `docs/specs.md` §6.3 (project layout, `Gravity.Dsl.MsBuild/`), §6.4 (Build integration), §7 phase 9; constitution architectural constraint "Build integration parity" and Principle VI (Pluggable, not prescriptive).
**Predecessor:** `specs/002-phase-8-versioning/spec.md` (Phase 8 — additive-only versioning enforcement, `--as-of`, `Foo@N` syntax). This document extends that surface; it does not replace it. Phases 4–7 (the four remaining reference emitters — JSON Schema, GraphQL, OpenAPI, AsyncAPI) remain future work and are NOT delivered by this phase. The sample emitter introduced here is deliberately a non-production "outline" emitter and is not a substitute for any of the four pending reference emitters.
**Scope statement:** This is the lockable spec for the narrowed Phase 9 slice agreed with the user: the MSBuild target packaged for `<PackageReference>` consumption, and a single reference **sample** emitter (`Gravity.Dsl.Emitter.Sample.Outline`) that exists to demonstrate the `IEmitter` contract end-to-end for third-party authors. The full Phase 9 description in `docs/specs.md` §7 ("Build integration, CLI ergonomics, deterministic output, error reporting; emitter authoring guide and reference sample") is deliberately split: the **emitter authoring guide** is deferred to a Phase 9b deliverable, **CLI ergonomics polish** (color, `--watch`, `--verbose`) is deferred, **full deterministic-output hardening** beyond what Phase 0–3 already shipped is deferred, and **error-reporting polish** beyond what Phase 0–3 / Phase 8 already shipped is deferred.

## 1. Goals (tied to constitution)

| Goal | Constitution principle |
|---|---|
| G-1. Domain authors consume the toolchain through ordinary .NET package management — a single `<PackageReference Include="Gravity.Dsl.MsBuild" Version="..." />` makes `.gravity` files participate in `dotnet build` without a global tool install, without checked-in `.targets`, and without bespoke per-repo scripting. | "Build integration parity"; VI. Pluggable, not prescriptive. |
| G-2. The MSBuild target is a thin entry point that delegates to the same compiler + emitter host as the `gravc` CLI. Identical input produces identical generated output and identical diagnostics regardless of entry point. The MSBuild path is not a re-implementation. | "Build integration parity" (architectural constraint, verbatim). |
| G-3. Diagnostics surface in MSBuild canonical format `path(line,col): error <ruleId>: <message>` so IDE click-through (Rider, Visual Studio, VS Code OmniSharp/C# Dev Kit) jumps to the offending span in the `.gravity` source. Severity maps faithfully: errors fail the build, warnings do not. | V. AI-readable (the same readable rule ids round-trip through tool surfaces); constitution Quality-standards "Error messages carry source location, the rule violated". |
| G-4. The reference sample emitter `Gravity.Dsl.Emitter.Sample.Outline` exists for one purpose: to be a copy-paste starting point for community emitter authors. Its surface is intentionally minimal (Markdown outline per entity), its repo location (`samples/emitters/outline/`) signals "sample, not reference", and its NuGet id is namespaced `.Sample.` so nobody mistakes it for a production target. | VI. Pluggable, not prescriptive (the pluggability story is only credible if a complete, end-to-end exemplar exists). |
| G-5. The `--as-of` deprecation-window flag from Phase 8 surfaces through MSBuild as a single property (`<GravityDslAsOf>`). This is the **only** CLI flag the MSBuild target re-exposes — every other behaviour is taken from `.gravity.config`. Adding more property bridges requires an explicit spec amendment so the MSBuild surface stays small. | "Build integration parity"; constitution amendment policy. |
| G-6. The MSBuild package is itself deterministic: `dotnet pack Gravity.Dsl.MsBuild` produces byte-identical `.nupkg` content for identical inputs (no timestamps in metadata, sorted file ordering inside the package). Generated `.cs`, `.md`, etc. files under `obj/` likewise remain byte-deterministic, inheriting the Phase 0–3 guarantee. | "Deterministic output" (architectural constraint). |
| G-7. The integration smoke test under `tests/integration/msbuild-smoke/` is the parity contract made executable: it consumes the published `Gravity.Dsl.MsBuild` and `Gravity.Dsl.Emitter.Sample.Outline` packages exactly as a consumer would, and CI fails if `dotnet build` against it does not produce the documented artifacts. | "Build integration parity"; constitution Quality-standards "Integration tests across the full reference emitter set, run on every PR". |

## 2. Non-goals (Phase 9, narrowed slice)

- **NG-1.** **Emitter authoring guide.** The end-user-facing prose document that teaches "how to write a new emitter" is a Phase 9b deliverable and is explicitly excluded from this slice. The sample emitter ships without an accompanying tutorial; community authors read its source and the existing `IEmitter` XML doc until 9b lands.
- **NG-2.** **CLI ergonomics polish.** `--watch`, `--verbose`, coloured output, progress reporting, summary tables, and the existing `gravc` surface beyond what Phase 0–3 / Phase 8 shipped are out of scope. The CLI is not modified by Phase 9 except as required to expose code paths the MSBuild target consumes.
- **NG-3.** **Roslyn source generator.** Deferred per `docs/specs.md` §9 "source-generator vs CLI-first". A source generator would run in-process inside the C# compiler and is a separate (and larger) architecture problem; the MSBuild target runs `gravc` (or its in-proc equivalent `CompilerPipeline`) as a `BeforeBuild` step, which is sufficient for v1 and is the explicit plan in `docs/specs.md` §6.4.
- **NG-4.** **Watch mode / live regeneration.** No file-system watcher. Generation happens once per `dotnet build` invocation.
- **NG-5.** **IDE / project-system hooks.** No special integration with VS / Rider / VS Code beyond what MSBuild itself provides through its standard logger. IDE click-through happens because the diagnostic format is the MSBuild-canonical one (G-3 / FR-240), not because we ship a project-system extension.
- **NG-6.** **Per-build telemetry.** No phone-home, no opt-in usage stats, no build-time pings.
- **NG-7.** **Full deterministic-output hardening pass.** The constitution-level determinism bar is already met by Phase 0–3 emitters (AC-6a / AC-6b) and Phase 8 ordering rules. Phase 9 inherits that bar; it does not add new determinism guarantees beyond G-6 (deterministic `.nupkg` packaging).
- **NG-8.** **Error-reporting polish.** Beyond the format mapping in FR-240 / FR-241, the diagnostic content (rule id, message, span) is unchanged from Phase 0–3 / Phase 8. Re-wordings, fix hints, structured codes, MSBuild help-link URLs, etc. are deferred.
- **NG-9.** **Phases 4–7 reference emitters.** JSON Schema, GraphQL, OpenAPI, AsyncAPI remain TBD. The outline sample emitter does **not** substitute for any of them; it is not a production target.
- **NG-10.** **Everything in Phase 8 NG-1..NG-7 still applies.** Cross-compile version diffing via a persisted manifest, per-field `@deprecated`, runtime guards, semantic-default-change detection, auto-migration tooling, JSON Schema/GraphQL/OpenAPI/AsyncAPI emitters, Registry features, AI authoring tooling, and cross-namespace diffing edge cases remain out of scope.

The package contains `.props`, `.targets`, and the compiler + emitter-host + reference-emitter binaries needed to run generation. No global tool install required; no checked-in `.targets`. | NuGet is the lingua franca of .NET build composition; `buildTransitive/` ensures a domain library that pulls in the DSL package pulls the generation step too. | LD-12. Sample emitter naming | NuGet package id `Gravity.Dsl.Emitter.Sample.Outline`. Repo location `samples/emitters/outline/Gravity.Dsl.Emitter.Sample.Outline/`. Target name `outline`. The `.Sample.` segment in the package id is permanent and signals "sample, not reference" wherever the package surfaces (NuGet.org listings, IDE references, log lines). | The repo-layout signal (`samples/emitters/outline/`) covers in-repo readers; the package-id signal covers downstream consumers who only see the NuGet listing. ## 4. Functional requirements

Levels: **MUST**, **SHOULD**, **MAY**. Each requirement traces to a constitution principle (Roman numeral) or to the named architectural constraint. Phase 9 numbering starts at FR-200 to make rule cross-references unambiguous against Phase 0–3 (FR-001..FR-098) and Phase 8 (FR-100..FR-161). New rule ids introduced by this phase use the `MSB001..MSB010` range for MSBuild-host errors; FR-225..FR-228 reserve the lower end as needed.

### 4.1 MSBuild item type and target

- **FR-200 (MUST, "Build integration parity", VI).** A new MSBuild item type `<GravityDsl>` is recognised by the imported `.targets`. When the consuming csproj declares no explicit `<GravityDsl>` items, the target injects a default `<GravityDsl Include="**/*.gravity" />` so the canonical "drop a `.gravity` file next to your csproj and rebuild" workflow works without configuration. Consumers may opt out of the default by setting `<GravityDslDisableDefaultItems>true</GravityDslDisableDefaultItems>`; this is the documented escape hatch for projects that want fully-explicit item lists. The implicit-include glob runs **after** explicit `<GravityDsl Remove="..." />` items so a consumer can subtract files from the default set without disabling it entirely.
- **FR-201 (MUST, "Build integration parity").** The generation target runs as a `BeforeBuild` dependency by default. The target name is `GravityDslGenerate`. Consumers SHOULD NOT need to override the hook in normal use; for projects with unusual build graphs the property `<GravityDslHook>BeforeCompile</GravityDslHook>` selects an earlier hook (legal values: `BeforeBuild` (default), `BeforeCompile`, `BeforeResolveReferences`). Any other value produces `MSB006` (configured hook not recognised) and fails the build.
- **FR-202 (MUST, "Build integration parity", VI).** Each `<GravityDsl>` item supports two pieces of metadata that override the .gravity.config defaults for that item only: `Output` overrides the emitter-output directory (resolved relative to `$(MSBuildProjectDirectory)`); `Emitter` is a semicolon-separated whitelist of emitter target names to run for this item (equivalent to the CLI `--emitter <name>` flag, repeatable). Item metadata applies per-item — if a consumer has two `<GravityDsl>` items pointed at different directories with different `Emitter` whitelists, each item's run honours its own metadata. When item metadata is absent, the `.gravity.config` block for the emitter wins.
- **FR-203 (MUST, "Build integration parity", VI).** Config discovery follows three rules in order, first match wins: (a) explicit `<GravityDslConfig>` MSBuild property naming an absolute or `$(MSBuildProjectDirectory)`-relative path to a `.gravity.config` file; (b) a `.gravity.config` next to the consuming csproj; (c) no config — falls back to the same default behaviour the CLI uses (`CompilerPipeline.LoadConfigs` synthesises one entry per registered emitter with `output: <TargetName>` — see `CompilerPipeline.cs` lines 162-178). When the explicit property names a file that does not exist, `MSB003` fires; when option (b) is used and the file is malformed, the existing `Gravity.Dsl.Emitter.ConfigLoader` diagnostics surface verbatim through the MSBuild logger.
- **FR-204 (MUST, "Build integration parity").** Emitter invocation is parallel and matches CLI semantics: the target invokes `EmitterHost.Run` (the same entry the CLI calls from `CompilerPipeline.Gen`, line 116) which runs each enabled emitter on its own task. Per-item parallelism (multiple `<GravityDsl>` items running concurrently) is **not** introduced — items are processed serially; the parallelism unit is the emitter set within a single pipeline run. This matches the CLI's `Gen` shape exactly.
- **FR-205 (MUST, "Build integration parity").** Exit semantics match the CLI: any `DiagnosticSeverity.Error` in the diagnostics returned by `CompilerPipeline.Gen` causes the MSBuild task to fail (`Log.HasLoggedErrors == true` after the run, MSBuild fails the build). Warnings (`DiagnosticSeverity.Warning`) and informational diagnostics do not fail the build. This mirrors `Program.cs`:`RunGen` (lines 56-69) which returns `result.Success ? 0 : 1`.
- **FR-206 (MUST, V).** Severity mapping: `DiagnosticSeverity.Error` → MSBuild error (`Log.LogError`); `DiagnosticSeverity.Warning` → MSBuild warning (`Log.LogWarning`); `DiagnosticSeverity.Info` → MSBuild message at `MessageImportance.Low`. The rule id from the diagnostic flows into the MSBuild error/warning code field so IDE filtering and `MSBuild /warnaserror:VAL025` style escalation works out of the box.
- **FR-207 (MUST, V, "Error messages").** Diagnostic formatting through the MSBuild logger uses the canonical MSBuild form `path(line,col): error <ruleId>: <message>`. The path is the resolved `.gravity` source path (matching what the existing `DiagnosticFormatter.Format` already produces). When the offending diagnostic has no useful source span (host-level rules `HOST001`, `HOST002`, `MSBxxx`), the path field is the consuming csproj path and the line/column default to `(1,1)`. The behaviour is implemented by reusing the existing `DiagnosticFormatter` from `Gravity.Dsl.Cli` so the CLI and MSBuild surfaces emit byte-identical message text.
- **FR-208 (MUST, I, VI).** The `GravityDslGenerate` target declares MSBuild incremental-build keys: `Inputs="@(GravityDsl)"`, `Outputs="$(GravityDslOutputDir)/.gravity-stamp"`. The target's final step `Touch`es the stamp file. On a subsequent `dotnet build` against unchanged sources, MSBuild skips the target entirely (no codegen, no log output). When any `.gravity` source has a newer mtime than the stamp file, the target re-runs in full. The stamp file is registered with `<FileWrites>` so `dotnet clean` removes it alongside the generated tree.

### 4.2 NuGet packaging

- **FR-210 (MUST, "Build integration parity", VI).** The `Gravity.Dsl.MsBuild` NuGet package ships exactly the following layout. The build-asset directory is `buildTransitive/` only — modern NuGet (5.0+, every .NET SDK since 3.0) handles `buildTransitive/` for both direct and transitive consumers correctly, so the legacy `build/` mirror is intentionally omitted:
  - `buildTransitive/Gravity.Dsl.MsBuild.props` — declares `<GravityDsl>` item type metadata, default property values (`<GravityDslDisableDefaultItems>`, `<GravityDslHook>`, `<GravityDslAsOf>`, `<GravityDslConfig>`), and imports `Gravity.Dsl.MsBuild.targets`.
  - `buildTransitive/Gravity.Dsl.MsBuild.targets` — declares `GravityDslGenerate` target hooked to the configured stage and a single `UsingTask` for the in-package MSBuild task assembly.
  - `tasks/net9.0/` — the MSBuild task assembly plus the compiler, emitter host, AST, CLI (`gravc.dll`), and reference emitter assemblies needed to invoke generation. The directory is `tasks/net9.0/` (the build-task NuGet convention), not `tools/net9.0/` (which is for `dotnet tool install`). The tasks tree contains the closed dependency set of `CompilerPipeline.Gen` so the package is self-contained against a SDK with the pinned `global.json` version. The C# reference emitter (`Gravity.Dsl.Emitter.CSharp`) is bundled because it is the Phase 0–3 reference target; sample / Phase 4–7 emitters are **not** bundled — they are separate NuGet packages (LD-12 for the outline emitter; future packages for Phases 4–7). The InProc default (FR-230) loads the task assembly and the bundled `gravc.dll` from `tasks/net9.0/`; no separate `gravc.exe` is required in the package contents for the default flow.
- **FR-211 (MUST, "Build integration parity").** The package targets `net9.0` (matching the rest of the toolchain pinned in `global.json` and `Directory.Build.props`). Imports succeed in csproj projects targeting `net9.0` or any newer net*x.0 the package is forward-compatible with. The `.targets` file uses no MSBuild SDK feature newer than what ships with the `global.json` pin (`9.0.314`) so importing from a `net9.0` consumer never trips an SDK-version error.
- **FR-212 (MUST, "Deterministic output").** Packaging is deterministic: `dotnet pack Gravity.Dsl.MsBuild -c Release` produces a `.nupkg` whose entries are sorted by path under ordinal comparison and whose internal timestamps are zeroed via the SDK's `ContinuousIntegrationBuild` / `Deterministic` properties (already enabled repo-wide by `Directory.Build.props` lines 11-12). Running `dotnet pack` twice in succession on a clean tree produces byte-identical `.nupkg` files; this is asserted by an integration test (AC-9.7-pack).
- **FR-213 (SHOULD, VI).** The package declares `<DevelopmentDependency>true</DevelopmentDependency>` and `<IncludeBuildOutput>false</IncludeBuildOutput>` so the tooling assemblies under `tasks/net9.0/` do not leak into the consumer's `bin/` output as ordinary references. Consumers receive build behaviour, not runtime references.

### 4.3 Sample emitter (`Gravity.Dsl.Emitter.Sample.Outline`)

- **FR-220 (MUST, VI).** The sample emitter implements `IEmitter` in a new project `samples/emitters/outline/Gravity.Dsl.Emitter.Sample.Outline/`. `TargetName` is `"outline"`. `AnnotationNamespace` is `"outline"` (claimed exclusively; FR-052 / `HOST002` enforces uniqueness across the emitter set — if another emitter ever claimed `outline`, host startup fails). `SupportedAstVersions` is `">=1.0.0 <2.0.0"`, mirroring the C# reference emitter so a Phase 8 AST bump to `1.1.0` does not break it.
- **FR-221 (MUST, VI, V).** The emitter walks `model.Declarations` in `(FQN ordinal, Version ascending)` order (the existing Phase 8 FR-161 ordering) and writes exactly one `.md` file per `EntityDecl`. Value types (`ValueTypeDecl`) and enums (`EnumDecl`) are deliberately not rendered — keeping the surface small reinforces the sample-not-reference framing (LD-12). Each entity file contains six sections, in this fixed order: `## Identity`, `## Relations`, `## Properties`, `## Lifecycle`, `## Events`, `## Commands`. Empty sections render the heading followed by `_(none)_` so the output always carries all six headings (this keeps the diff-against-baseline check trivial and the structure self-documenting for first-time readers).
- **FR-222 (MUST, "Deterministic output").** The emitter's output is byte-deterministic across runs. Iteration inside each section follows the Phase 0–3 ordering already established by the AST node order (relations in declaration order, properties in declaration order, lifecycle states in declaration order, etc.). No timestamps, no host machine name, no working-directory paths appear in the rendered Markdown. The file header (if any) follows the same shape as `CSharpFileFormatter` produces for the C# emitter — a single deterministic line naming the source `.gravity` file by `Path.GetFileName` (Phase 0–3 `CSharpEmitter.GravityRelativePath`, lines 129-140), then a blank line, then the six sections.
- **FR-223 (MUST, VI).** Configuration schema: exactly one key, `output` (string, required, no default). No other keys are accepted; an unrecognised key under the `outline:` block in `.gravity.config` fires the existing `Gravity.Dsl.Emitter.ConfigLoader` "unknown key" diagnostic verbatim. Minimal config is intentional — every additional knob is a thing a copy-paste author has to understand before they can extend it.
- **FR-224 (MUST, VI).** The sample ships as a **separate** NuGet package `Gravity.Dsl.Emitter.Sample.Outline` (per LD-12). It is **not** bundled into `Gravity.Dsl.MsBuild` (per FR-210). Consumers who want Markdown output add a second `<PackageReference Include="Gravity.Dsl.Emitter.Sample.Outline" Version="..." />` to their csproj. The two packages cooperate at MSBuild-import time: the outline emitter package contains a `buildTransitive/Gravity.Dsl.Emitter.Sample.Outline.props` that contributes its DLL path to a shared MSBuild item `<GravityDslEmitterAssembly>` collected by the main package's `GravityDslGenerate` target. This is the documented extension model for third-party emitters and the sample is the canonical exemplar.
- **FR-225 (MUST, VI).** Repo location is `samples/emitters/outline/Gravity.Dsl.Emitter.Sample.Outline/`. The `samples/` ancestor is the visual signal in repo browsing; the `.Sample.` segment in the package id (LD-12) is the equivalent signal in NuGet listings. Both layers are intentional: a future "production Markdown emitter" — if one ever ships — would live under a project root sibling to `Gravity.Dsl.Emitter.CSharp/` and would carry a non-`.Sample.` NuGet id.

### 4.4 Build-integration parity

- **FR-230 (MUST, "Build integration parity", VI).** The MSBuild target and the CLI share a single set of binaries: the same `Gravity.Dsl.Compiler.dll`, `Gravity.Dsl.Ast.dll`, `Gravity.Dsl.Emitter.dll`, `Gravity.Dsl.Emitter.CSharp.dll`, and (for the in-proc path) `gravc.dll` (the CLI assembly, whose `AssemblyName` is `gravc` per its csproj) ship in both `gravc`'s install tree and `Gravity.Dsl.MsBuild`'s `tasks/net9.0/`. The MSBuild task implementation invokes `CompilerPipeline.Gen` directly (in-proc) by default; an `<GravityDslExecMode>OutOfProc</GravityDslExecMode>` property switches the task to spawn the bundled `gravc` executable instead. The default is in-proc; out-of-proc is the documented escape hatch for environments where the MSBuild task's `AssemblyLoadContext` collides with the consumer's (e.g. Roslyn version drift — see plan.md §6 risk register). The public surface of `Gravity.Dsl.Cli` (`CompilerPipeline.Check`, `CompilerPipeline.Gen`, `PipelineResult`, `DiagnosticFormatter`, `CliRuleIds`) is treated as a stable contract per LD-13. Future changes to it follow the same additive-only / deprecation discipline as the AST contract (Principle VI).
- **FR-231 (MUST, "Build integration parity").** Identical exit semantics. The MSBuild task returns `false` (failure) when the pipeline returns `Success == false` (i.e. any `DiagnosticSeverity.Error` is present); returns `true` (success) otherwise. Warnings do not fail the build. This pins MSBuild's behaviour to `Program.cs:RunGen` (lines 56-69) line-for-line.
- **FR-232 (MUST, "Build integration parity").** Identical diagnostic content. Rule ids (`PARSE001..PARSE020`, `LEX001..LEX002`, `RES001..RES006`, `VAL001..VAL030`, `HOST001..HOST002`, `CFG004`, `CLI001..CLI002`, plus the new `MSB001..MSB010` from this phase) flow through unchanged. The message body is byte-identical to the CLI surface. The source span is byte-identical. AC-9.5 asserts this against a fixture.
- **FR-233 (MUST, "Build integration parity", IV).** The Phase 8 `--as-of` flag surfaces as MSBuild property `<GravityDslAsOf>` accepting `YYYY-MM-DD`. Default behaviour matches the CLI default: when the property is absent or empty, the MSBuild task computes `DateOnly.FromDateTime(DateTime.UtcNow)` **at task entry** (the constitutional restriction that `DateTime.UtcNow` must live only in CLI-level entry points is honoured because the MSBuild task assembly is a CLI-equivalent entry point; the `BannedSymbols.txt` allowlist is extended to cover the MSBuild task project under the same convention as `Gravity.Dsl.Cli`). A malformed `<GravityDslAsOf>` value emits `MSB001` (rule id reserved here) and fails the build before any compilation work runs, mirroring `CLI002` from Phase 8 (FR-142). This is the **only** MSBuild-property → CLI-flag bridge introduced by this phase; documenting that constraint here is itself part of the spec so future amendments are explicit. The defaulted-to-today behaviour means: running the same `dotnet build` on two different days against the same sources can produce **different** diagnostics (VAL030 may fire on day N+1 if a `until` date crossed midnight). CI consumers requiring strict reproducibility MUST pin `<GravityDslAsOf>` to an explicit date. The MSBuild incremental-build mechanism (FR-208 / Inputs+Outputs) does NOT solve this on its own: when the stamp is fresh but the as-of date crosses a deprecation boundary, the user must manually delete the stamp or change a source. This is acknowledged as an inherent property of date-driven validation; documenting it is the mitigation.
- **FR-234 (MUST, "Build integration parity", VI).** The MSBuild task package's NuGet content contains `tasks/net9.0/gravc.dll` (the CLI assembly, named per its `AssemblyName`), not `Gravity.Dsl.Cli.dll`. AC-9.1's expected-entry list reflects this. Co-located alongside `gravc.dll` under `tasks/net9.0/` are the MSBuild task assembly, the compiler / AST / emitter closure, and the bundled `gravc` executable used by the OutOfProc mode (FR-230). If a Phase 9 minor extension to the CLI helper surface is required to support per-item Output/Emitter overrides via the MSBuild task — specifically an overload of `CompilerPipeline.Gen` that accepts an explicit `IList<string>` of source files instead of scanning `inputRoot` — that overload ships as an additive public method on `CompilerPipeline` and inherits the LD-13 stability contract from day one.
- **FR-235 (MUST, "Build integration parity").** **Negative parity coverage.** AC-9.5 fixture pair: same `.gravity` input is run through (a) `gravc gen` invoked directly, and (b) `dotnet build` against a csproj consuming `Gravity.Dsl.MsBuild`. The generated artifact tree is hashed (SHA-256 over file-paths-and-contents, sorted) and the two hashes must match. Any divergence fails the test — there is no allowlist of "expected differences", because by FR-230 there are none.

### 4.5 Diagnostics format and reserved rule ids

- **FR-240 (MUST, V, "Error messages").** The MSBuild logger formats every diagnostic as `<path>(<line>,<column>): <severity> <ruleId>: <message>` where `<severity>` is `error`, `warning`, or `info`. This is the format MSBuild's standard console logger, binary log viewer, and IDE consumers all natively parse for click-through. The format is identical to what `Gravity.Dsl.Cli.DiagnosticFormatter.Format` already produces (path is colon-separated in the CLI, parenthesised in the MSBuild logger — the only intentional divergence, motivated by IDE click-through; both forms convey the same data). The MSBuild path-form is canonical because it is the form IDEs jump to; the CLI path-form is preserved unchanged for compatibility with existing CI scrapers.
- **FR-241 (MUST, V).** Severity mapping (formal restatement of FR-206): `DiagnosticSeverity.Error` → `Log.LogError(subcategory: null, errorCode: ruleId, ...)`; `DiagnosticSeverity.Warning` → `Log.LogWarning(subcategory: null, warningCode: ruleId, ...)`; `DiagnosticSeverity.Info` → `Log.LogMessage(MessageImportance.Low, ...)` (no rule-id flow through; informational diagnostics do not have a code surface in MSBuild and the `Info` severity is rare enough that the loss is acceptable). The `errorCode` / `warningCode` parameters carry the rule id so `MSBuild /warnaserror:VAL025` works without any custom configuration on the consumer side.
- **FR-242 (MUST, V).** New rule ids reserved by Phase 9, all in the `MSB` (MSBuild host) namespace:
  - `MSB001` — `<GravityDslAsOf>` malformed (CLI002 equivalent for the MSBuild surface).
  - `MSB002` — `<GravityDslExecMode>` unrecognised value.
  - `MSB003` — `<GravityDslConfig>` names a file that does not exist on disk.
  - `MSB004` — `<GravityDsl>` item resolved to zero files on disk and `<GravityDslDisableDefaultItems>` was true (likely misconfiguration; the build does not fail without items present, but a warning surfaces under this rule when the inclusion is non-empty in source but empty after resolution).
  - `MSB005` — A `<GravityDsl>` item's `Output` metadata resolves outside `$(MSBuildProjectDirectory)` (rooted path or `..` escape). Mirrors `CFG004` from Phase 0–3 (FR-097) but applies at the MSBuild item layer before the host-level pre-flight, so the error message is targeted to the item line rather than to the config file.
  - `MSB006` — `<GravityDslHook>` set to a value not in `{BeforeBuild, BeforeCompile, BeforeResolveReferences}`.
  - `MSB007` — Task tree under `tasks/net9.0/` missing an expected assembly (defence-in-depth against a corrupted package; should never fire in practice).
  - `MSB008..MSB010` — Reserved for forward use; not consumed by this slice.

### 4.6 Determinism and ordering (inherited)

- **FR-250 (MUST, "Deterministic output").** Phase 9 introduces no new ordering rules. The MSBuild target invokes the same `CompilerPipeline.Gen` that the CLI invokes (FR-230); diagnostic ordering follows Phase 0–3 / Phase 8 rules (FR-160 / FR-161). The MSBuild logger preserves the order it receives diagnostics in.
- **FR-251 (MUST, "Deterministic output").** Generated artifact paths under `obj/Generated/<emitter>/...` are sorted by `BufferedEmitterOutput` (the existing Phase 0–3 buffered sink) before being flushed to disk. Re-running `dotnet build` on an unchanged tree produces identical mtimes (within filesystem-rounding tolerance) and identical file contents.

## 5. Acceptance criteria (Phase 9 "done")

- **AC-9.1.** **Package shape and `dotnet pack` round-trip.** `dotnet pack Gravity.Dsl.MsBuild -c Release` produces `Gravity.Dsl.MsBuild.<version>.nupkg`. Unzipping the `.nupkg` shows exactly: `buildTransitive/Gravity.Dsl.MsBuild.props`, `buildTransitive/Gravity.Dsl.MsBuild.targets`, `tasks/net9.0/*.dll` (closed dependency set of `CompilerPipeline.Gen` including the MSBuild task assembly, `Gravity.Dsl.Compiler.dll`, `Gravity.Dsl.Ast.dll`, `Gravity.Dsl.Emitter.dll`, `Gravity.Dsl.Emitter.CSharp.dll`, `gravc.dll` (the CLI assembly, named per its `AssemblyName`; FR-234), plus the `Pidgin`, `YamlDotNet`, `System.CommandLine`, and `Microsoft.CodeAnalysis.CSharp.Workspaces` runtime dependencies pinned in `Directory.Packages.props`), and the standard NuGet `.nuspec` + `[Content_Types].xml`. `Microsoft.Build.Utilities.Core.dll` MUST NOT appear under `tasks/net9.0/`: MSBuild's own `AssemblyLoadContext` provides its own copy of that assembly, and bundling it would produce ALC collision warnings; the bundled-out exclusion is enforced via `ExcludeAssets="runtime"` on the `PackageReference` (plan.md §3.5). No other files. A separate `dotnet pack Gravity.Dsl.Emitter.Sample.Outline -c Release` produces `Gravity.Dsl.Emitter.Sample.Outline.<version>.nupkg` containing `buildTransitive/Gravity.Dsl.Emitter.Sample.Outline.props` (contributing to `<GravityDslEmitterAssembly>`) and `tasks/net9.0/Gravity.Dsl.Emitter.Sample.Outline.dll`.
- **AC-9.2.** **End-to-end MSBuild generation.** A test csproj under `tests/integration/msbuild-smoke/` declaring `<PackageReference Include="Gravity.Dsl.MsBuild" Version="..." />` and `<GravityDsl Include="**/*.gravity" />` produces generated `.cs` under `obj/Generated/csharp/` and (when the outline package is also referenced) generated `.md` under `obj/Generated/outline/` on `dotnet build`. The generated C# compiles as part of the same build (the smoke csproj sets `<EnableDefaultCompileItems>true</EnableDefaultCompileItems>` and includes `obj/Generated/csharp/**/*.cs` in its own compilation). Build exit code is `0`.
- **AC-9.3.** **Syntax error fails the build with click-through diagnostic.** A fixture under `tests/integration/msbuild-smoke-broken/` containing a `.gravity` file with a deliberate syntax error (e.g. `entity Foo version 1 { properties { x: ; } }`) causes `dotnet build` to fail (exit code non-zero). The MSBuild log shows exactly one error in the canonical form `Foo.gravity(<line>,<col>): error PARSE<NNN>: <message>` where the rule id matches what the CLI would emit for the same input. The file path is the resolved absolute path to the offending `.gravity` source so IDE click-through jumps to it.
- **AC-9.4.** **Warnings surface as MSBuild warnings, do not fail build.** A fixture exercising `VAL025` (transition removed; Phase 8 SHOULD-warning) builds successfully (exit `0`) but the MSBuild log includes exactly one warning in the canonical form `Foo.gravity(<line>,<col>): warning VAL025: <message>`. The warning carries `VAL025` in the MSBuild `warningCode` field so `MSBuild /warnaserror:VAL025` would escalate it to an error in a consumer's build (a follow-up assertion verifies that escalated mode does fail the build).
- **AC-9.5.** **CLI / MSBuild byte-for-byte parity (FR-235).** A parity test runs the canonical `samples/registry/` input through (a) `gravc gen --input samples/registry --output gen-cli --emitter csharp --emitter outline --as-of 2026-05-18` and (b) `dotnet build tests/integration/msbuild-smoke/MsBuildSmoke.csproj /p:GravityDslAsOf=2026-05-18` (which writes to `obj/Generated/`), then SHA-256-hashes the merged tree of `(relative-path, content-bytes)` tuples from each run. The two hashes are equal. Any divergence fails the test.
- **AC-9.6.** **Outline emitter section coverage.** Running the outline emitter against the Phase 0–3 canonical `TimeEntry` fixture produces a `TimeEntry.md` file containing all six sections in the documented order — `## Identity`, `## Relations`, `## Properties`, `## Lifecycle`, `## Events`, `## Commands` — with each non-empty section listing the underlying DSL nodes (relations enumerated by `name: Target [cardinality]`, properties by `name: Type`, lifecycle by states list followed by transitions list, events by `EventName(field: Type, ...)`, commands by `CommandName(arg: Type, ...) returns ReturnType (side_effect: EventName)`). An empty section renders the heading followed by `_(none)_` (no listing).
- **AC-9.7.** **Outline emitter byte-determinism.** A golden-file test under `tests/golden/outline/` byte-checks the emitted `.md` for the three canonical entities (`Employee.md`, `TimeEntry.md`, `Project.md`). The emitter runs twice in a single process and produces byte-identical files; the same suite on a second OS image in CI produces byte-identical files. This is the outline counterpart to Phase 0–3 AC-6b.
- **AC-9.7-pack.** **Package determinism.** Running `dotnet pack Gravity.Dsl.MsBuild -c Release` twice on a clean tree produces byte-identical `.nupkg` files. The same applies to `dotnet pack Gravity.Dsl.Emitter.Sample.Outline -c Release`.
- **AC-9.8.** **External consumer adds the outline emitter via package reference.** A second test csproj under `tests/integration/msbuild-smoke-outline-only/` declares `<PackageReference Include="Gravity.Dsl.MsBuild" Version="..." />` AND `<PackageReference Include="Gravity.Dsl.Emitter.Sample.Outline" Version="..." />`, plus a `.gravity.config` listing only the `outline` emitter (no `csharp` block). On `dotnet build`, `obj/Generated/outline/Employee.md` is produced; `obj/Generated/csharp/` is **not** created (the `csharp` emitter is registered with the host but its `.gravity.config` block is absent, so it is disabled — matching the existing `Gravity.Dsl.Emitter.ConfigLoader` semantics).
- **AC-9.9.** **`<GravityDslAsOf>` plumbed end-to-end.** Setting `<GravityDslAsOf>2027-01-01</GravityDslAsOf>` on the smoke fixture's csproj (with a `.gravity` source carrying `deprecates version 1 until "2026-12-31"`) fires `VAL030` (deprecation window expired; Phase 8 FR-138) at MSBuild error severity and fails the build. Setting `<GravityDslAsOf>2026-12-31</GravityDslAsOf>` against the same fixture does **not** fire `VAL030` (boundary semantics from Phase 8 AC-8.10 hold through the MSBuild path). Setting `<GravityDslAsOf>2026-13-45</GravityDslAsOf>` produces `MSB001` and fails the build before any compilation work runs.
- **AC-9.10.** **Annotation namespace ownership enforced across the MSBuild + sample-emitter pair.** A test exercises the FR-052 / `HOST002` path through the MSBuild entry point: a stub second emitter that also claims `"outline"` is loaded alongside `Gravity.Dsl.Emitter.Sample.Outline`. The MSBuild build fails with `HOST002` naming both claimants, in canonical MSBuild diagnostic format. No generated output is produced. This pins the cross-phase contract that the existing host-startup rule survives the journey to the MSBuild surface.
- **AC-9.11.** **Item metadata overrides config defaults.** A fixture sets `<GravityDsl Include="domain/*.gravity" Output="custom/out" Emitter="outline" />` while its `.gravity.config` declares `outline: { output: obj/Generated/outline }`. The build writes to `custom/out/`, not to `obj/Generated/outline/`, and only the `outline` emitter runs even though `csharp` is also registered. This pins FR-202.
- **AC-9.12.** **Hook selection.** Setting `<GravityDslHook>BeforeCompile</GravityDslHook>` re-orders the target so `GravityDslGenerate` runs after `BeforeBuild` but before the Roslyn `Compile` target. A trace through `MSBuild /bl` confirms the target ordering. Setting `<GravityDslHook>Garbage</GravityDslHook>` produces `MSB006` and fails the build before compilation.
- **AC-9.13.** **Empty input.** A consuming csproj with `<GravityDslDisableDefaultItems>true</GravityDslDisableDefaultItems>` and zero `<GravityDsl>` items builds successfully (exit `0`) and produces no `obj/Generated/**` tree. No `MSB004` fires because the items list is empty by configuration, not by glob misfire.
- **AC-9.14.** **No global tool dependency.** The MSBuild flow does not require `gravc` installed as a global tool. A test asserts `dotnet tool list --global` does NOT contain `gravc` in the test environment before the smoke build runs; the smoke build still succeeds. This validates the `<PackageReference>`-only distribution model (LD-9) without relying on PATH-scrubbing tautologies.
- **AC-9.15.** **Incremental build.** Two back-to-back `dotnet build` invocations against an unchanged smoke fixture: the first build runs `GravityDslGenerate` (a log line confirms target execution); the second build skips it (no log line, target reports up-to-date via the `Inputs`/`Outputs` mechanism declared in FR-208). A test under `tests/integration/msbuild-smoke-incremental/` pins this with two consecutive in-process `dotnet build` invocations plus `dotnet build /verbosity:minimal` log scraping. Touching any `.gravity` source between the two builds (updating its mtime) restores full execution on the second build.
