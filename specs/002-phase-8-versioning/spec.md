# Gravity DSL — Specification (Phase 8: Additive-only versioning enforcement)

**Status:** Locked for implementation
**Date:** 2026-05-18
**Driven by:** `docs/specs.md` §4.4 (Versioning), §7 phase 8 ("Versioning checks: additive-only enforcement, deprecation flags, version-pair comparison"); constitution Principle IV (Additive-only versioning by default).
**Predecessor:** `specs/001-gravity-dsl/spec.md` (Phases 0–3 — compiler core, AST publication, C# emitter). This document extends that surface; it does not replace it.
**Scope statement:** This is the lockable spec for Phase 8 of `docs/specs.md` §7: additive-only enforcement, deprecation flags, version-pair comparison, and the version-qualified type reference syntax (`Foo@N`) that makes multi-version coexistence well-defined. Phases 4–7 (the four remaining reference emitters) ship after Phase 8 per LD-8 below; Phase 9–10 (build integration, OSS launch) remain future work.

---

## 1. Goals (tied to constitution)

| Goal | Constitution principle |
|---|---|
| G-1. The compiler refuses to emit when a newer entity or value-type version makes a breaking change against a prior in-scope version unless a valid `deprecates` chain authorizes it. Generated artifacts can never silently shed a field, narrow a type, or drop a state. | IV. Additive-only versioning by default. |
| G-2. The full breaking-change taxonomy from Principle IV (field removal, type narrowing, lifecycle state removal, command removal, semantic-changing default changes) is machine-detected; nothing in the taxonomy is left to reviewer discretion. | IV. Additive-only versioning by default. |
| G-3. Multi-version coexistence is permitted only through an explicit, machine-verifiable chain: `entity Foo version N deprecates version N-1 until "<date>"`. A newer version without that chain re-triggers `RES004` (Phase 0–3 behaviour) so authors cannot accidentally fork an entity into incompatible siblings. | IV; III (the read-only contract must remain unambiguous downstream). |
| G-4. Version-qualified type references (`Foo@N`) make the chosen version explicit at call sites; unqualified references continue to resolve to the maximum declared version. The grammar extension is additive (only a new optional suffix on `NamedTypeRef`). | V. AI-readable; VI. Pluggable, not prescriptive. |
| G-5. The deprecation-window check is a function of an injected `currentDate` (`DateOnly`), not of `DateTime.UtcNow`. The compiler stays pure; the CLI injects today's date (or the operator-supplied `--as-of`). Builds remain deterministic and reproducible from a single CLI invocation forward. | IV; constitution architectural constraint "Deterministic output". |
| G-6. AST changes for Phase 8 are strictly additive on existing records (one new nullable field on `NamedTypeRef`); `AstVersion` stays in the `1.x` line. No third-party emitter built against `1.0.0` breaks. | VI. Pluggable, not prescriptive; architectural constraint "Stable AST contract". |
| G-7. Phase 8 ships before Phases 4–7. The four remaining reference emitters are written against a versioning-aware AST so they never have to be retrofitted later. | architectural constraint "Stable AST contract"; risk note in `docs/specs.md` §8 ("Versioning model tested late"). |

## 2. Non-goals (Phase 8)

- **NG-1.** Cross-compile version diffing via a persisted manifest of "what was emitted last time". Phase 8 only diffs versions that coexist in the **current** input set. Deferred to Phase 9+.
- **NG-2.** Per-field `@deprecated` annotations marking individual fields as scheduled for removal at the next major version. The whole-entity `deprecates` clause is the only deprecation surface in Phase 8. Deferred to Phase 9+.
- **NG-3.** Runtime guards (e.g., a generated check that refuses to deserialize a dropped field). Phase 8 is compile-time only, consistent with the constitution out-of-scope discipline ("Runtime DSL evaluation").
- **NG-4.** Semantic-default-change detection. The v1 grammar has no field defaults; this taxonomy entry from Principle IV is parked until defaults exist. The compiler does not pretend to detect changes it cannot see.
- **NG-5.** Auto-migration tooling (rewrite v1 callers to `Foo@1`, etc.). Out of scope.
- **NG-6.** Everything in `specs/001-gravity-dsl/spec.md` NG-1..NG-7 still applies. JSON Schema/GraphQL/OpenAPI/AsyncAPI emitters, MSBuild integration, LSP, formatter, Registry features, and AI authoring tooling remain out of scope for this phase.
- **NG-7.** Cross-namespace diffing edge cases where the same simple name appears under different FQNs. Phase 8 diffs **strictly within a single FQN**; two entities with the same simple name but different namespaces are not version-pair candidates.

## 3. Locked decisions (user-approved)

| Decision | Value | One-line rationale |
|---|---|---|
| LD-5. Multi-version coexistence | Lift `RES004` **only** when the newer version carries `deprecates version <N> until "<date>"` naming the previous version (chained). Without the chain, `RES004` still fires. Resolver keys declarations by `(FQN, Version)`. | Multi-version coexistence is a domain-modelling necessity (callers migrating off v1 need v1 to still resolve), but a free-for-all forks the entity. The chain is the explicit, machine-checkable signal. |
| LD-6. Version-qualified type refs | `Foo@N` syntax lands in Phase 8. AST `NamedTypeRef` gains `int? Version`. Unqualified `Foo` resolves to the maximum declared in-scope version. Validator emits an error if `@N` references a non-existent version. | Without this, a property typed `Project` is ambiguous once `Project@1` and `Project@2` coexist. Defaulting unqualified to max preserves the Phase 0–3 reading. |
| LD-7. As-of date for deprecation window | New CLI flag `--as-of YYYY-MM-DD`, defaulting to `DateOnly.FromDateTime(DateTime.UtcNow)` **inside the CLI driver only**. The compiler library exposes `Validator.Validate(model, claimedAnnotationNamespaces, currentDate)`. | Keeps the compiler pure and deterministic; lets CI reproduce yesterday's diagnostics. Today's date does not leak into golden-file goldens. |
| LD-8. Phase sequencing | Phase 8 ships **before** Phases 4–7. AST additions here remain backward-compatible (new nullable fields on existing records, new optional argument on validator entry point). `AstVersion` stays `1.x`. | Retrofitting four emitters with versioning awareness after the fact is the cited risk in `docs/specs.md` §8. Doing the AST shape once, up-front, eliminates it. |

## 4. Functional requirements

Levels: **MUST**, **SHOULD**, **MAY**. Each requirement traces to a constitution principle (Roman numeral). Phase 8 numbering starts at FR-100 to make rule cross-references unambiguous against the Phase 0–3 spec (FR-001..FR-098).

### 4.1 Grammar extensions

- **FR-100 (MUST, IV, V).** The `NamedTypeRef` production accepts an optional `@<positive int>` suffix: `Foo@2`, `Money@1`, `ApprovalResult@3`. The suffix is parsed before the optional `?` and `[]` modifiers, e.g. `Foo@2?`, `Foo@2[]`, `Foo@2[]?`. The suffix is **not** legal on relation targets (already constrained to single entity names by FR-022) — when seen there, the parser emits `PARSE020`. The suffix is also illegal on primitive types (`Int@2`); the parser emits `PARSE020`. The suffix is also illegal in command `returns <Name>` clauses in Phase 8: the current AST represents the return type as a bare identifier (`CommandDecl.ReturnsType: string`), and upgrading to a full `TypeRef` is out of scope for Phase 8. The parser emits `PARSE020` if `@N` is seen after `returns`.
- **FR-101 (MUST, V).** A malformed `@N` suffix — missing integer, non-integer literal, zero or negative integer, leading `+`, leading zero (`@01`), or whitespace between `@` and the number — emits `PARSE020` at the `@` token. The parser recovers by treating the `@` as unconsumed input and continues with the unqualified type ref so a single malformed suffix does not cascade.
- **FR-102 (MUST, IV).** The grammar gains no other constructs in Phase 8. No new keywords. `deprecates`, `version`, and `until` were already reserved in FR-004; no further reserved-word changes.

### 4.2 AST surface (additive)

- **FR-110 (MUST, VI).** `NamedTypeRef` gains one new nullable property: `int? Version`. The C# `record` shape becomes `NamedTypeRef(string Name, bool IsOptional, bool IsArray, SourceSpan Span, int? Version = null)`. The property is `null` when the source had no `@N` suffix. **Placing `Version` at the END as an optional positional parameter with default `null` preserves the 4-arg constructor signature from `1.0.0`. Source code that constructed `new NamedTypeRef(name, false, false, span)` against `1.0.0` continues to compile and runs identically. The only construction-site change is internal to the parser, which passes the version explicitly. Pattern-matching with 4-positional deconstruction (`is NamedTypeRef(var n, var o, var a, var s)`) is no longer arity-compatible — it must be rewritten to use named patterns; this is the trade-off for keeping ctor compatibility. A repo-wide grep confirms zero such positional deconstruction sites exist today.**
- **FR-111 (MUST, VI).** `AstVersion.Value` is bumped to `"1.1.0"` (minor revision, additive). The AST package documents that consumers compiled against `1.0.0` continue to function because the new positional parameter has a default value: a `1.0.0` consumer that did `new NamedTypeRef(name, isOpt, isArr, span)` still resolves to the new 5-parameter primary constructor with `Version = null` filled in by the default. A consumer that pattern-matches by named properties (the only style used internally — confirmed via grep) is unaffected.
- **FR-112 (MUST, VI).** No new top-level AST records are introduced in Phase 8. No existing record's existing members change shape or position. This protects Phase 0–3 third-party emitters.
- **FR-113 (SHOULD, VI).** The AST package's XML doc on `DeprecatesClause` is amended to point to FR-130..FR-138 (this phase) as the consumer of `Version` and `UntilIso8601`.

### 4.3 Resolver changes

- **FR-120 (MUST, IV).** The resolver keys declarations by `(FQN, Version)`, not by FQN alone. The declaration map type changes from `ImmutableSortedDictionary<string, TopLevelDecl>` to `ImmutableSortedDictionary<DeclKey, TopLevelDecl>` where `DeclKey = (string Fqn, int Version)` with ordinal/numeric comparison. Iteration order remains stable and deterministic (`Fqn` ordinal, then `Version` ascending).
- **FR-121 (MUST, IV).** Two declarations sharing `(FQN, Version)` still emit `RES004` (unchanged Phase 0–3 behaviour: duplicate declaration of the same version is always an error).
- **FR-122 (MUST, IV).** Two declarations sharing FQN but with different `Version` are allowed **only** when the higher-versioned declaration carries `deprecates version <prev> until "<date>"` and `<prev>` is one of the lower versions in the same FQN's declaration set. If multiple older versions coexist, the chain must cover the immediately-preceding version (`Vmax-1`); skipping versions (e.g. v3 deprecates v1 while v2 also exists undeprecated) emits `VAL027`.
- **FR-123 (MUST, IV).** When no `deprecates` chain exists between two coexisting versions of the same FQN, the resolver re-fires `RES004` with the existing Phase 0–3 message ("entity '<FQN>' is declared more than once; multi-version coexistence requires a deprecates chain"). The message body is updated; the rule id is preserved so existing tooling that filters on `RES004` keeps working.
- **FR-124 (MUST, IV).** A `deprecates version <N>` whose `<N>` does not match any declared version of the same FQN emits `VAL028` ("deprecates version N references no declared version of <FQN>"). This is a validator rule, not a resolver rule, because the resolver may not yet have completed the version index when an individual file is parsed; it is checked at the validation pass on the merged model.
- **FR-125 (MUST, IV).** A `deprecates version <N>` clause where `<N>` is greater than or equal to the declaring entity's own version emits `VAL029` ("entity <FQN>@<self> may not deprecate version <N>; <N> must be strictly less than <self>"). Self-deprecation and forward-deprecation are both rejected.
- **FR-126 (MUST, IV).** Per LD-6, unqualified `NamedTypeRef` with `Version == null` resolves to the maximum declared version of the matching FQN **reachable from the current file's imports**. Formally: `max(VersionIndex[fqn] ∩ ImportedFqns(fileImports[fpath]))` — the highest version of the FQN that is declared in the current file itself or in a transitively imported file. A v2 declared elsewhere in the model but **not** transitively imported by file F is invisible to unqualified refs in F. The resolver attaches the resolved version to its internal lookup table; the AST node is **not** mutated (the source-written form is preserved for round-trip).
- **FR-127 (MUST, IV).** `NamedTypeRef.Version` set to `N` resolves to the declaration `(FQN, N)`. If no such declaration exists in scope, the resolver emits `RES003` with a message distinguishing missing-version from missing-name ("type '<Name>@<N>' is not declared; <Name> exists with versions {<list>}" vs "name '<Name>' is not defined or imported in this scope"). This satisfies the constitution Quality-standards bullet on missing-import vs missing-definition distinction, extended to missing-version.

### 4.4 Breaking-change detection (validator)

Phase 8 introduces eight diagnostic rules. Each runs **per FQN pair** `(Vprev, Vnext)` where `Vnext` is the immediately-higher version and a `deprecates` chain ties `Vnext` to `Vprev`. If no chain exists, FR-123 already rejected the input at the resolver layer; the validator's breaking-change pass does not run.

- **FR-130 (MUST, IV).** **VAL020 — field removed.** For every entity property name, value-type field name, event payload field name, and command argument name present in `Vprev`, the validator checks that `Vnext` declares a field of the same name in the corresponding container. A missing name emits `VAL020` with message `"<container>.<name> was removed in <FQN>@<Vnext>; field removal is a breaking change (requires explicit removal via a future @deprecated field annotation, deferred to Phase 9+)"`. Container is one of `entity-property`, `value-type-field`, `event-payload`, `command-argument`, identified by entity-event-or-command name where applicable.
- **FR-131 (MUST, IV).** **VAL021 — type narrowed.** For every same-named field surviving from `Vprev` to `Vnext`, the validator checks the type pair for narrowing. The primitive narrowing table is **closed-form**: a primitive transition is narrowing iff the pair `(prev, next)` appears in the table below (a pair not listed is **not** narrowing — this is the inverse of the previous "fallthrough narrows" semantics and matches the constitution's bias toward permissive widening):

  | Prev → Next | Narrowing? | Note |
  |---|---|---|
  | Decimal → Int | yes | precision lost |
  | Decimal → Long | yes | precision lost |
  | Long → Int | yes | range lost |
  | DateTime → Date | yes | time-of-day lost |
  | String → Int | yes | non-text representation |
  | String → Long | yes | non-text representation |
  | String → Decimal | yes | non-text representation |
  | String → Boolean | yes | non-text representation |
  | String → UUID | yes | structural narrowing of free text |
  | String → Date | yes | structural narrowing |
  | String → DateTime | yes | structural narrowing |
  | Same Kind → Same Kind | no | no narrowing |
  | Int → Long, Long → Decimal, Int → Decimal, Date → DateTime | no | explicit widenings |
  | UUID → String, Date → String, DateTime → String, Int → String, Long → String, Decimal → String, Boolean → String | no | broadening to free text — never flagged |
  | Any other primitive pair not listed above | no | fallthrough is NOT narrowing |

  In addition to the primitive table above, the following structural narrowings apply:
  - Optionality is lost: `T?` → `T` (`IsOptional` true→false).
  - Array-ness is lost: `T[]` → `T` (`IsArray` true→false).
  - A `NamedTypeRef.Version` decreases (e.g. property typed `Money@2` becomes `Money@1`); this is treated as narrowing because callers depending on `Money@2`'s shape lose surface.
  - A `NamedTypeRef.Name` changes (a "rename" is functionally a removal plus an addition; emit `VAL020` on the old name and `VAL021` is **not** emitted in addition).
  - Cross-kind transitions (primitive ↔ named): treated as narrowing because the named ref's shape is unknown to the diff engine and the safest assumption per Principle IV is "this is a contract change".

  The message is `"<container>.<name>: type narrowed from <prev> to <next> in <FQN>@<Vnext>"`. The "from" / "to" forms render the full surface (`String?[]`, `Money@2`, etc.) so the fix is obvious. The closed-form rule is provably complete by enumerating cases in the test matrix (AC-8.2).
- **FR-132 (MUST, IV).** **VAL022 — lifecycle state removed.** Every state name in `Vprev.lifecycle.states` must appear in `Vnext.lifecycle.states`. Missing states emit `VAL022` with message `"lifecycle state '<state>' removed from <FQN>@<Vnext>; state removal is a breaking change"`. Adding new states is allowed.
- **FR-133 (MUST, IV).** **VAL023 — command removed.** Every command name in `Vprev.commands` must appear in `Vnext.commands`. Missing commands emit `VAL023` with message `"command '<name>' removed from <FQN>@<Vnext>; command removal is a breaking change"`.
- **FR-134 (MUST, IV).** **VAL024 — event removed.** Every event name in `Vprev.events` must appear in `Vnext.events`. Missing events emit `VAL024`.
- **FR-135 (SHOULD, IV).** **VAL025 — transition removed (warning).** Every `(From, To, OnEvent)` triple in `Vprev.lifecycle.transitions` SHOULD appear in `Vnext`. Missing triples emit `VAL025` at **warning** severity. Rationale for warning-not-error: a transition is a narrowing of a state graph, not a contract field; removing `Submitted -> Approved on Approved` while keeping the involved states and event leaves type-shape callers unaffected but does change runtime behaviour. The author may have intentionally tightened the lifecycle. The warning surfaces the case; if and when runtime guards land (Phase 9+), this is the rule to promote to error.
- **FR-136 (MUST, IV).** **VAL026 — command argument breaking change.** For every command surviving from `Vprev` to `Vnext`, the validator runs four sub-checks. All emit `VAL026` with a sub-cause appended to the message: (a) argument removed: any `arg.Name` in `Vprev` missing from `Vnext`; (b) argument renamed: detected as removal-plus-addition (VAL026 fires on the removal under sub-cause `"argument removed"`; the addition is also checked under (d)). **When the new (renamed) argument's type is required (`TypeRef.IsOptional == false`), the rename emits two VAL026 diagnostics — one with sub-cause `"argument removed"` on the old name and one with sub-cause `"required argument added"` on the new name. When the new argument's type is optional, only the removal fires.** (c) argument type narrowed: same closed-form narrowing table as VAL021 (FR-131); (d) new required argument added: any `arg.Name` in `Vnext` not present in `Vprev` whose type is not optional (`TypeRef.IsOptional == false`). Sub-cause messages: `"argument removed"`, `"argument type narrowed from <prev> to <next>"`, `"required argument added"`. **Note on return types.** Because `CommandDecl.ReturnsType` is a bare identifier in the v1 AST (no `Version` slot), VAL021 over the return slot detects only Name changes; a rename of the return type's underlying entity registers as a `CommandDecl.ReturnsType` string change which VAL023 reports as command-removed-plus-added when the return-type identifier differs. Type-version narrowing of the return slot is not detectable in Phase 8 and is deferred to Phase 9+ along with the AST extension to `TypeRef`. A `with side_effect <event>` change is checked under VAL024 when the new event does not exist.
- **FR-137 (MUST, IV).** **VAL027 — deprecates chain broken.** When two coexisting versions `Vmax > Vmid > ...` exist for the same FQN, every intermediate version must be either present in a chain (`Vk+1 deprecates version Vk`) or absent from the in-scope declaration set. A coexisting `Vmid` that is **not** referenced by `Vmid+1`'s `deprecates` clause emits `VAL027` ("deprecates chain broken: <FQN>@<Vmid> coexists with <FQN>@<Vmid+1> but is not named in its deprecates clause"). This complements FR-123 (which catches the simpler "no chain at all" case) by catching skipped links in a longer chain.
- **FR-138 (MUST, IV).** **VAL030 — deprecation window expired.** For every `DeprecatesClause` on every declaration in the resolved model, the validator parses `UntilIso8601` (well-formedness already enforced by `VAL009` in Phase 0–3) and compares it against the injected `currentDate` (`DateOnly`, see LD-7). If `UntilIso8601 < currentDate`, emit `VAL030` ("deprecation window for <FQN>@<Vprev> expired on <date>; remove the deprecated version or extend the window"). The check is **strict less-than**: on `currentDate == UntilIso8601` the window is still open through end-of-day; this matches the intuitive reading of `until "<date>"` and gives users a clean cutover boundary. **Example.** Source `deprecates version 1 until "2026-12-31"`. With `currentDate = 2026-12-31`, no VAL030 fires (window is still open through end of day). With `currentDate = 2027-01-01`, VAL030 fires. This matches the colloquial reading of `until <date>` — the deprecation window ends after the named date, not before it.

### 4.5 CLI surface (compiler-driven, validator entry point)

- **FR-140 (MUST, IV).** `Validator.Validate` gains a third required parameter `DateOnly currentDate`. Callers in the test suite pass deterministic values; the CLI passes the operator-supplied or computed value. The signature becomes `Validate(ResolvedModel model, IReadOnlyCollection<string> claimedAnnotationNamespaces, DateOnly currentDate)`.
- **FR-141 (MUST, IV).** The `gravc` CLI gains a `--as-of YYYY-MM-DD` flag. When absent, the CLI computes `DateOnly.FromDateTime(DateTime.UtcNow)` inside its `Main` (never inside the compiler library). The default is logged in verbose mode so reproducible builds can capture and re-supply it.
- **FR-142 (MUST, V).** A malformed `--as-of` value emits a CLI-layer error (`CLI002`) before any compilation work begins. The error names the bad value and the expected format. This is a CLI-only rule and is not part of the `Gravity.Dsl.Compiler` rule namespace.

### 4.6 Diagnostic shape

- **FR-150 (MUST, V).** Every Phase 8 diagnostic carries (a) a stable rule id from `{PARSE020, RES003-extended, RES004-extended, VAL020..VAL030}`, (b) the source span of the offending construct, (c) a message that names the FQN and the version(s) involved, (d) where the constitution Quality-standards "include the fix where possible" bar can be met, a one-clause hint. Example: `VAL020` on a removed field on `Employee@2` appends "; the prior version 'Employee@1' declared this field — keep it, mark it `?`, or add a `deprecates` chain and a future `@deprecated` field annotation when that lands".
- **FR-151 (MUST, V).** All Phase 8 rule ids are added to `RuleIds.cs`. Constants follow the existing naming pattern (`Val020 = "VAL020"`, etc.). `PARSE020` lives in the parser's rule-id surface, mirroring `PARSE010` from Phase 0–3.

### 4.7 Determinism and ordering

- **FR-160 (MUST, VI).** Diagnostics from the breaking-change pass are emitted in a deterministic order: (1) ordered by FQN ordinal, (2) then by `Vnext` ascending, (3) then by rule id ordinal, (4) then by source span (file path ordinal, line, column). This matches the existing Phase 0–3 diagnostic ordering bar and is required so golden-file tests of the validator output stay stable across runs and platforms.
- **FR-161 (MUST, VI).** The `(FQN, Version)`-keyed declaration map iterates in `(Fqn ordinal, Version ascending)` order to keep all downstream emitter output stable when the same FQN has multiple in-scope versions.

## 5. Acceptance criteria (Phase 8 "done")

- **AC-8.1.** **Field removal detected.** A two-file fixture declaring `Employee version 1 { properties { hireDate: Date; manager_id: UUID; } }` and `Employee version 2 deprecates version 1 until "2099-01-01" { properties { hireDate: Date; } }` produces exactly one `VAL020` diagnostic on `Employee@2`, naming `manager_id`. The same fixture without the `deprecates` clause produces `RES004` and **no** `VAL020` (the breaking-change pass does not run without a chain).
- **AC-8.2.** **Type narrowing detected (each row in the narrowing table).** A parameterised test runs nine fixtures, one per narrowing case (`T?`→`T`, `T[]`→`T`, `Decimal`→`Int`, `Long`→`Int`, `String`→`Int`, `String`→`UUID`, `DateTime`→`Date`, `Money@2`→`Money@1`, `Foo`→`Bar`). Each fixture produces exactly the expected `VAL021` (or `VAL020` for the rename case), with the rendered "from"/"to" forms matching the source.
- **AC-8.3.** **Widening allowed.** A fixture narrowing `Int` → `Long` (i.e. widening) produces zero `VAL021` diagnostics.
- **AC-8.4.** **State and transition rules.** A fixture removing one state from `Vprev` to `Vnext` produces exactly one `VAL022`. A fixture removing only a transition (states and event preserved) produces exactly one `VAL025` at warning severity, not an error.
- **AC-8.5.** **Command and event removal.** Separate fixtures removing a command and removing an event produce `VAL023` and `VAL024` respectively. Renaming a command is reported as `VAL023` on the removal (the addition is not flagged unless it introduces a required-argument issue under VAL026).
- **AC-8.6.** **Command argument matrix.** Four fixtures cover VAL026 sub-causes: argument removed, argument renamed (reported as removed), argument type narrowed, new required argument added. Each produces the expected `VAL026` with the matching sub-cause text. A new **optional** argument added produces zero diagnostics.
- **AC-8.7.** **Multi-version coexistence with valid chain.** A fixture with `Employee@1`, `Employee@2 deprecates version 1 until "2099-12-31"`, and `Employee@3 deprecates version 2 until "2099-12-31"` resolves without `RES004` and runs the breaking-change pass twice: `(1, 2)` and `(2, 3)`. Diagnostics for each pair are independent.
- **AC-8.8.** **Broken chain rejected.** A fixture with `Employee@1`, `Employee@2` (no deprecates), `Employee@3 deprecates version 2` produces (a) `RES004` for the `(1, 2)` pair (missing chain) and (b) `VAL027` is not double-counted on the same pair. A separate fixture with `Employee@1`, `Employee@2 deprecates version 1`, `Employee@3 deprecates version 1` (skipped link — v3 should deprecate v2) produces `VAL027`.
- **AC-8.9.** **Deprecates target validation.** Fixtures producing `VAL028` (deprecates points to non-existent version) and `VAL029` (deprecates self-reference or forward reference) each produce exactly the named rule and no false `VAL027`.
- **AC-8.10.** **Deprecation window enforcement is date-driven and deterministic.** The same fixture validated with `currentDate = 2026-05-18` and `UntilIso8601 = "2026-05-17"` produces `VAL030`; validated with `currentDate = 2026-05-17` (equal) it produces zero `VAL030`. **Year-boundary case**: with `UntilIso8601 = "2026-12-31"` and `currentDate = 2026-12-31`, zero `VAL030`; with the same `until` and `currentDate = 2027-01-01`, one `VAL030`. The compiler library is invoked directly in the test — no clock injection at the library layer.
- **AC-8.11.** **CLI `--as-of` plumbed.** `gravc gen --input samples/registry --output gen --emitter csharp --as-of 2099-01-01` exits 0 against a fixture whose `deprecates ... until "2026-12-31"` would normally still be in window today, then exits non-zero when the same command is run without `--as-of` and the system date is past 2026-12-31. A second test invokes the same CLI with `--as-of 2026-13-45` and expects `CLI002` with a non-zero exit.
- **AC-8.12.** **Version-qualified type refs.** `properties { payment: Money@2; }` parses to `NamedTypeRef(Name: "Money", Version: 2, ...)`. `properties { payment: Money@2?[] }` parses with both modifiers preserved. Malformed `Money@`, `Money@01`, `Money@-1`, `Money@ 2` each emit `PARSE020`. `properties { count: Int@2; }` emits `PARSE020` (no version suffix on primitives). `relations { paid_by: Employee@2 ... }` emits `PARSE020` (no version suffix on relation targets, per FR-100).
- **AC-8.13.** **Unqualified resolves to max.** A fixture with `Project@1` and `Project@2` in scope and a property `lead_project: Project;` resolves to `Project@2` in the resolver's lookup. Adding `lead_project_legacy: Project@1;` resolves to `Project@1`. A property `lead_project: Project@5;` with no declared `Project@5` emits the missing-version variant of `RES003` naming the available versions.
- **AC-8.13b.** **Unqualified resolves to max within imports.** A two-file fixture: `a.gravity` imports only `defs_v1.gravity` (declaring `Project version 1`); `defs_v2.gravity` exists at the same input root declaring `Project version 2 deprecates version 1 until "2099-12-31"` but is **not** imported by `a.gravity`. An unqualified `Project` reference in `a.gravity` resolves to `Project@1` (not `@2`), because `Project@2` is not transitively reachable from `a.gravity`'s import set. This pins the FR-126 "in imported scope" semantics against the alternate "max in model" reading.
- **AC-8.14.** **AST stability.** `AstVersion.Value == "1.1.0"`. A Phase 0–3 third-party emitter built and assembly-loaded against `1.0.0` is loaded successfully by the emitter host (the host's compatibility check accepts the previous minor); a regression test pins this behaviour.
- **AC-8.15.** **Diagnostic ordering.** A fixture exercising VAL020, VAL021, VAL022, VAL023, VAL024, VAL025, VAL026, VAL027, VAL028, VAL029, VAL030 in a single resolved model produces diagnostics in the documented order (FR-160). A golden file under `tests/golden/diagnostics/phase8/` is byte-checked.
- **AC-8.16.** **Round-trip preserves `@N`.** Parsing a source with `Money@2`, re-emitting via the source emitter, and re-parsing produces an AST with `NamedTypeRef.Version == 2`. The Phase 0–3 round-trip test harness (AC-1 in the predecessor spec) is extended with two fixtures covering version-qualified type refs. **The Phase 0–3 AC-1 invariant continues to hold**: every existing fixture under `tests/fixtures/parser/` and every file under `samples/registry/` round-trips through parse → SourceWriter → parse with no new structural inequality. The `SpanIgnoringEquality` helper is extended to compare `NamedTypeRef.Version`, and the existing fixtures all parse to `Version == null`, so the new comparison is satisfied trivially.
