# Gravity DSL — Specification (Phase 5: PostgreSQL DDL reference emitter)

**Status:** Locked for implementation
**Date:** 2026-05-20
**Driven by:** `docs/specs.md` §5 (Pluggable emitter architecture), §5.2 (Reference emitters shipped with v1), §6.3 (project layout, `Gravity.Dsl.Emitter.*/`), §7 phase 5 ("PostgreSQL DDL reference emitter"); constitution Principle VI (Pluggable, not prescriptive — dominant), Principle II (Domain-only — boundary against storage hints sneaking back), Principle IV (Additive-only versioning — migrations are derivable, not authored), and the "Deterministic output" architectural constraint.
**Predecessor:** `specs/004-phase-4-json-schema-emitter/spec.md` (Phase 4 — JSON Schema reference emitter). The JSON Schema emitter's project layout, `IEmitter` plug-in surface, `EmitterRegistry.Discover` discovery contract, `EmitterConfigSchema` configuration validation pipeline, banned-APIs analyzer scope, deterministic-output bar, and golden-file test pattern are inherited verbatim. Phase 8 versioning (`specs/002-phase-8-versioning/spec.md`) is also a hard predecessor — the per-entity `version` integer is what migrations key off (FR-420).
**Scope statement:** This is the lockable spec for Phase 5 of `docs/specs.md` §7: a NuGet-distributed PostgreSQL DDL reference emitter (`Gravity.Dsl.Emitter.PostgresDdl`) that consumes the existing AST and emits two coordinated artifact sets per `.gravity` source: (1) a **baseline schema** of idempotent table creation SQL (`schema/`), and (2) a **per-version migration ledger** for entity-version evolution (`migrations/`). The target PostgreSQL schema (the database namespace, e.g. `public`, `hr_prod`, `tenant_42`) is configurable. Relations emit foreign-key columns plus btree / GIN indexes sized for the query plans the cardinality implies. Phases 6 / 7 (GraphQL SDL, OpenAPI, AsyncAPI) remain future work. The Gravity Registry layer (releases, library imports, scopes, permissions) remains out of scope.

## 1. Goals (tied to constitution)

| Goal | Constitution principle |
|---|---|
| G-1. Every byte of generated DDL — every `CREATE TABLE`, every `CREATE TYPE`, every `CREATE INDEX`, every `ALTER TABLE` in the migration ledger — is derived from `.gravity` source. No hand-authored `.sql` is admitted into any sample, fixture, or downstream consumer; the emitter is the single source of those bytes. Operators who currently hand-edit `schema.sql` migrate to a build that fails when the emitted file diverges from the source of truth. | I. The DSL is the spec. |
| G-2. The emitter declares no storage-layout, runtime-topology, or deployment vocabulary in its **input** (the DSL grammar is untouched) and only the minimum operationally-necessary vocabulary in its **output** (column names, foreign-key constraints, btree / GIN indexes derived from cardinality). Partitioning, tablespaces, RLS policies, row-level triggers, stored procedures, custom collations, statistics targets, fillfactor knobs, and any deployment-specific tuning are explicitly out of scope (NG-1..NG-7) and any future request to surface them in the DSL must be refused — they belong in a deployment manifest layer that composes DDL with environment policy. | II. Domain-only. |
| G-3. Generated DDL describes the **structural** contract only: column names, types, nullability, primary keys, foreign keys, and indexes that derive directly from relation cardinality. Downstream code (application servers, ORMs, query layers) consumes the schema as read-only; it cannot add columns or relax constraints without re-running the emitter from updated `.gravity` source. Migration files are likewise read-only — they materialise diffs the DSL grammar already authorised through its additive-only rule (Principle IV) and cannot insert breaking changes the source did not declare. | III. Read-only at build time. |
| G-4. Per-entity version evolution produces a **migration ledger** with one file per version: `V1__<Entity>.sql` for the baseline (the entity at its lowest version in scope), `V2__<Entity>.sql`, `V3__<Entity>.sql`, ... for each subsequent version. Each migration carries `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` / `ADD CONSTRAINT IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` only — no `DROP`, no `ALTER COLUMN TYPE`, no narrowing — which the DSL's additive-only rule guarantees is sufficient. The Phase 8 breaking-change pass (`VAL020..VAL030`) is the upstream gate; if it admits a model, this emitter can produce a forward-only migration without further analysis. | IV. Additive-only versioning by default. |
| G-5. Generated DDL is readable by current LLMs and by domain reviewers without exotic tooling. SQL keyword case is consistent (uppercase keywords, lowercase identifiers — FR-450), column ordering follows DSL declaration order (FR-452), and the emitted files carry a header comment naming the source FQN and version so a reviewer reading `schema/hr/Employee.sql` understands which `.gravity` declaration to consult. The translation from "system X's schema" into "canonical Gravity entity" → "PostgreSQL DDL" remains a tractable round-trip for a human or an LLM. | V. AI-readable (extended to "ecosystem-readable" for the DDL target). |
| G-6. The PostgreSQL DDL emitter is **one plugin among many**: it is discovered through the existing `EmitterRegistry` scan, it claims `postgres` exclusively (FR-052 / `HOST002` enforces), it declares a stable `IEmitter` contract version, and it ships as a separate NuGet package (`Gravity.Dsl.Emitter.PostgresDdl`) — never bundled into the compiler core nor into `Gravity.Dsl.MsBuild`. A consumer enables it by adding a `<PackageReference>` next to `Gravity.Dsl.MsBuild`, mirroring the Phase 4 JSON Schema emitter's two-package layout (LD-12). | VI. Pluggable, not prescriptive. |
| G-7. The emitter runs and emits without any Registry artifact present. Registry concerns — scopes, permission grants, release manifests, library imports — do not appear in the generated DDL. The target-schema config key (LD-18) is deliberately a single-value override (`schema = "public"` by default) rather than a per-scope mapping: per-scope DDL deployment is a Registry concern and lives one layer up. | VII. Composable upward. |
| G-8. Output is byte-deterministic across runs and platforms: column ordering, `CREATE INDEX` ordering, migration-file path ordering, table-name → file-name mapping, LF newlines, UTF-8 no BOM, no timestamps in headers, no `gen_random_uuid()`-style non-deterministic defaults emitted by the emitter itself (a user-supplied `@postgres(default: "gen_random_uuid()")` is allowed because the user authored it deliberately — FR-441 vendor-extension contract; the emitter never *invents* a default). | "Deterministic output" (architectural constraint). |

## 2. Non-goals (Phase 5)

- **NG-1. Other database dialects.** MySQL, Microsoft SQL Server, Oracle, SQLite, CockroachDB, YugabyteDB, and the Postgres-wire-compatible derivatives (Aurora-postgres in flavour-incompatible modes, AlloyDB) are out of scope. Phase 6+ may introduce sibling emitters (`Gravity.Dsl.Emitter.MySqlDdl`, etc.); each will be a separate spec, a separate NuGet, and a separate target name. The Phase 5 emitter does **not** abstract over dialects — it emits PostgreSQL 14+ DDL only (LD-18).
- **NG-2. Row-level security (RLS).** No `CREATE POLICY`, no `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`. Tenancy-scoped access policy belongs in the Gravity Registry or in a downstream policy emitter; baking it into the DDL emitter would couple the DSL to a specific authorisation model.
- **NG-3. Triggers, stored procedures, functions, materialised views.** No `CREATE TRIGGER`, no `CREATE FUNCTION`, no `CREATE PROCEDURE`, no `CREATE MATERIALIZED VIEW`. Event sourcing, projection layout, and computed materialisation are downstream consumer concerns (Principle II).
- **NG-4. Partitioning, tablespaces, fillfactor, statistics targets, custom collations.** Performance tuning specific to a deployment topology is the consumer's concern. The emitter does not expose `PARTITION BY`, `TABLESPACE`, `WITH (fillfactor=...)`, `ALTER TABLE ... SET STATISTICS`, or `COLLATE`. A future deployment-manifest layer may overlay these on the emitted DDL; the emitter itself is topology-agnostic.
- **NG-5. Backward / down migrations.** No `DROP TABLE`, no `ALTER TABLE ... DROP COLUMN`, no `ALTER COLUMN ... TYPE` narrowing in any emitted artifact. The DSL's additive-only rule (Principle IV) makes down migrations not auto-derivable; consumers who need rollback compose hand-authored counter-migrations outside the emitter's surface.
- **NG-6. Connection strings, credentials, role grants.** No `CREATE ROLE`, no `GRANT`, no `\connect`, no environment-variable interpolation. The emitter produces DDL files; how, when, and by whom they are applied to a live database is a deployment concern (typically `psql -f`, `flyway migrate`, `sqitch deploy`, or a CI step). The emitter is **not** a migration runner.
- **NG-7. Events / Commands persistence.** The DSL declares `events { ... }` and `commands { ... }` per entity, but those are runtime constructs (event-sourced messages and RPC arguments respectively). The PostgreSQL DDL emitter encodes the **entity-state shape only** — identity, properties, relations, lifecycle state — and does **not** emit event-log tables, command-history tables, outbox tables, or any other event-sourcing infrastructure. A future `Gravity.Dsl.Emitter.PostgresEventStore` emitter (separate spec, separate target name) could layer event storage on top; Phase 5 does not.
- **NG-8. JSON / JSONB columns for nested value types.** Value types declared at namespace scope (`type ContactInfo { ... }`) compile to **PostgreSQL composite types** (`CREATE TYPE`) per LD-21, not to `JSONB` columns. JSONB is a deliberate downstream choice some operators prefer; it is not the DSL's default. A future `@postgres(storage: "jsonb")` annotation may be added (this spec reserves the key but does not consume it — FR-441 reservation list); without that annotation, composite types are the only output.
- **NG-9. Schema-per-entity layout.** All emitted tables live in **one** PostgreSQL schema, configured via the `schema` config key (LD-18, default `"public"`). Per-entity schemas, per-namespace schemas, and multi-schema layouts are deferred; they would couple the emitter to a multi-tenant topology decision (Principle II) and are better placed in a deployment layer.
- **NG-10. Configurable indexing strategy.** Relations always produce btree (cardinality-one foreign key) or GIN (cardinality-many array column) indexes per FR-433 / FR-434. Operators who need hash, BRIN, or partial indexes use a downstream tuning layer or wait for the `@postgres(index: "<method>")` annotation (reserved, not consumed in v1 — FR-441).
- **NG-11. Cross-database foreign keys.** Foreign-key constraints target tables in the same configured `schema`. A relation whose target entity has been split into a separate emitter run / separate schema is not supported in v1 — the resolver guarantees the target FQN is in the model, and the emitter assumes the target table is in the same PG schema.
- **NG-12. Hand-tuneable output formatting.** No `indent_size`, no `keyword_case`, no `quote_identifiers` knobs. Determinism is the only formatting contract (FR-450..FR-454); knobs that compromise byte-stability are refused.
- **NG-13. Everything in Phase 0–3 / Phase 4 / Phase 8 / Phase 9 NGs continues to apply.** Registry features, LSP / formatter / linter, runtime DSL evaluation, storage backends *beyond* this emitter's PG DDL surface, broker bindings, AI authoring tooling, and the emitter authoring guide (Phase 9b) all remain out of scope.

## 3. Locked decisions (user-approved)

| Decision | Value | One-line rationale |
|---|---|---|
| LD-18. Target dialect | **PostgreSQL 14+** as the floor. Syntax features used: `CREATE TABLE ... IF NOT EXISTS`, `CREATE INDEX ... IF NOT EXISTS` (added in 9.5), `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` (added in 9.6), `DO $$ ... EXCEPTION WHEN duplicate_object THEN NULL; END $$` for idempotent `CREATE TYPE` (CREATE TYPE has no native `IF NOT EXISTS` clause), `gen_random_uuid()` from `pgcrypto` extension (referenced only via user-supplied annotation, never emitted as a default — FR-441). | PostgreSQL 14 is the lowest still-supported LTS that has every syntax feature the spec needs, and is the realistic floor for any 2026-era deployment. Lower-bound versions would require synthesised workarounds (`PERFORM ... WHERE NOT EXISTS ...` blocks for index creation) that bloat the output. Targeting a single version keeps the emitter, the tests, and the goldens lean; multi-version dispatch adds combinatorial cost the v1 audience does not need. |
| LD-19. Target name + annotation namespace | **TargetName** `"postgres-ddl"` (kebab-case, matches the `docs/specs.md` §5.3 example convention and the JSON Schema emitter's `"json-schema"` precedent). **AnnotationNamespace** `"postgres"` (underscore-and-letters identifier per FR-004 — annotation namespaces cannot contain hyphens). The namespace claims the keyword subset documented in FR-441. The split between target name (kebab) and annotation namespace (identifier) is identical to LD-17 for JSON Schema and is now the established repo pattern. | Target names follow the user-visible target-naming convention (kebab); annotation namespaces follow the DSL identifier rule. The split must be explicit because the two lexical classes are not interchangeable. The `postgres` namespace is reserved exclusively by this emitter and refuses to coexist with a second claimant under `HOST002`. |
| LD-20. Configuration schema | **Three keys.** `output` (`String`, `Required`, no default) — the relative output root, host-implicit per Phase 0–3. `schema` (`String`, `Required: false`, `Default: "public"`) — the target PostgreSQL schema name into which every emitted table lives. `migration_prefix` (`String`, `Required: false`, `Default: "V"`) — the migration filename prefix; default `V` matches the Flyway convention (`V1__Foo.sql`); `sqitch` users may override to e.g. `"deploy/"`. Any other config key is rejected by the existing `CFG001` (Phase 0–3 unknown-key warning), which the emitter does **not** override. | One required, two optional knobs — the minimum surface that lets a consumer point at a tenant schema and pick a migration tool convention without compromising determinism (NG-12). The `schema` value is validated against the PG identifier rules at emitter pre-flight; an invalid value produces `PG001` (FR-470). |
| LD-21. Namespace-scope value types and enums | **Value types → PostgreSQL composite types** (`CREATE TYPE <schema>.<TypeName> AS (...)`). **Enums → PostgreSQL enum types** (`CREATE TYPE <schema>.<EnumName> AS ENUM (...)`). Both wrapped in an idempotent `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL; END $$;` block because PG lacks `CREATE TYPE IF NOT EXISTS`. A future `@postgres(storage: "jsonb")` annotation may opt value-type columns into JSONB encoding; the keyword is reserved (FR-441) but not consumed in v1 — composite is the only output. | Composite types let columns reference the value type by name (`contact contact_info NOT NULL`) and let the database enforce the value-type's shape at the storage layer (Principle III at the DB tier). JSONB would also work and is friendlier to migrations, but it sacrifices structural enforcement — the emitter defaults to the stricter option and reserves the looser one for explicit opt-in. |
| LD-22. File layout | **Two trees, one shared namespace path.** `<output>/schema/<namespace-path>/<DeclName>.sql` carries the **idempotent baseline DDL** for the latest in-scope version of each declaration: entity tables, composite types, enum types. `<output>/migrations/<namespace-path>/<prefix><Version>__<EntityName>.sql` carries the **per-version migration** for each entity version (and each value-type version when value types are versioned — FR-420). The schema/ tree reflects the **terminal state**; the migrations/ tree reflects the **path to that state**. Re-running `gravc gen` on an unchanged source produces byte-identical output in both trees. Multi-version coexistence (Phase 8 `Foo@1` + `Foo@2`): the migrations/ tree contains one file per version; the schema/ tree contains `Foo.v1.sql` and `Foo.v2.sql` (FR-425, symmetric to FR-310's JSON Schema rule). | Two trees because they answer two different operator questions: "what does the database look like when I'm done applying everything?" (schema/) and "what statements do I run to get there from a fresh database?" (migrations/). Folding them into one tree (e.g. concatenating migrations into a single deploy script) would force the operator to choose between the two views and would lose the per-version diff granularity migration tools need. Keeping them parallel preserves both views and lets the consumer pick. |

## 4. Functional requirements

Levels: **MUST**, **SHOULD**, **MAY**. Each requirement traces to a constitution principle (Roman numeral) or to a named architectural constraint. Phase 5 numbering starts at FR-400 to keep cross-references unambiguous against Phase 0–3 (FR-001..FR-098), Phase 8 (FR-100..FR-161), Phase 9 (FR-200..FR-260), and Phase 4 (FR-300..FR-365). New rule ids introduced by this phase live in the `PG001..PG010` range for emitter-level diagnostics.

### 4.1 Project + IEmitter contract

- **FR-400 (MUST, VI).** A new project `Gravity.Dsl.Emitter.PostgresDdl/Gravity.Dsl.Emitter.PostgresDdl.csproj` is introduced as a sibling to `Gravity.Dsl.Emitter.JsonSchema/`. It targets `net9.0`, inherits `Directory.Build.props` repo-wide settings (deterministic build, banned-APIs analyzer scope, treat-warnings-as-errors), depends on `Gravity.Dsl.Ast` (via `Gravity.Dsl.Ast.csproj` ProjectReference) and `Gravity.Dsl.Emitter` (likewise), and is packed as a stand-alone NuGet package `Gravity.Dsl.Emitter.PostgresDdl`. The package is **not** bundled into `Gravity.Dsl.MsBuild` (per Phase 9 FR-210); consumers wanting PostgreSQL DDL output add a second `<PackageReference>` mirroring the JSON Schema emitter's pattern from Phase 4.
- **FR-401 (MUST, VI).** The emitter type `Gravity.Dsl.Emitter.PostgresDdl.PostgresDdlEmitter : IEmitter` declares: `TargetName = "postgres-ddl"`, `AnnotationNamespace = "postgres"`, `SupportedAstVersions = SemanticVersionRange.Parse(">=1.0.0 <2.0.0")`. The version range spans both Phase 0–3's `1.0.0` AST and Phase 8's `1.1.0` AST; `NamedTypeRef.Version` is consumed when present and ignored when `null`.
- **FR-402 (MUST, VI).** `ConfigurationSchema` declares exactly three keys per LD-20:
  - `output` (`ConfigValueKind.String`, `Required: true`, no default)
  - `schema` (`ConfigValueKind.String`, `Required: false`, `Default: "public"`)
  - `migration_prefix` (`ConfigValueKind.String`, `Required: false`, `Default: "V"`)
- **FR-403 (MUST, VI).** `IEmitter.Emit(model, config, sink)` walks `model.Declarations` in `(FQN ordinal, Version ascending)` order — the FR-161 contract pinned by `ResolvedModel.Declarations` being an `ImmutableSortedDictionary` — and dispatches:
  - `EntityDecl` → entity-table renderer (FR-420..FR-435) **and** entity-migration renderer (FR-420 / FR-426)
  - `ValueTypeDecl` → composite-type renderer (FR-421)
  - `EnumDecl` → enum-type renderer (FR-422)
  
  Each emitted file is handed to `IEmitterOutput.WriteFile` with a `/`-normalised relative path under either `schema/` or `migrations/` (LD-22). No file is emitted for `EventDecl` or `CommandDecl` — those are runtime constructs (NG-7).
- **FR-404 (MUST, VI).** Configuration ingestion uses the existing `Gravity.Dsl.Emitter.ConfigLoader`. The emitter publishes its `EmitterConfigSchema`; the host validates `.gravity.yaml` before invocation. The emitter projects `EmitterConfig.Values` into an internal typed config record (`PostgresDdlEmitterConfig.From(config, diags)`) mirroring the `JsonSchemaEmitterConfig.From` pattern.
- **FR-405 (MUST, VI).** The emitter is `sealed` with a public parameterless constructor — the `EmitterRegistry` discovery contract requires both. Renderer helper types are `internal sealed` to keep the public surface area minimal.

### 4.2 Schema generation (baseline `schema/` tree)

- **FR-420 (MUST, I, IV).** Each `EntityDecl` at its **latest in-scope version** produces a baseline DDL file at `<output>/schema/<namespace-path>/<EntityName>.sql` (single-version case) or `<output>/schema/<namespace-path>/<EntityName>.v<Version>.sql` (multi-version case — when `model.Declarations` contains two or more entries with the same `FQN` and different `Version` values, **every** schema file for that FQN carries the `.v<Version>` infix, symmetric to FR-310). The namespace path is the DSL namespace with `.` replaced by `/`. The file content has the canonical shape:
  ```
  -- Gravity DSL — generated by gravc (target: postgres-ddl)
  -- source: <EntityFQN> version <N>
  -- DO NOT EDIT — re-run the emitter to regenerate.

  CREATE TABLE IF NOT EXISTS <schema>.<entity_name> (
      <column-list, in DSL declaration order>,
      PRIMARY KEY (<identity-column-name>)
  );

  <foreign-key constraints, sorted ordinally by name>

  <index statements, sorted ordinally by name>
  ```
  The header comment is fixed (three lines, exactly as shown; no timestamp, no machine name); `<schema>` is the configured target schema (LD-20); table and column identifiers follow the naming rules in FR-450..FR-454.
- **FR-421 (MUST, I, II).** Each `ValueTypeDecl` at namespace scope produces `<output>/schema/<namespace-path>/<TypeName>.sql` (or `.v<Version>.sql` per the multi-version rule) of shape:
  ```
  -- Gravity DSL — generated by gravc (target: postgres-ddl)
  -- source: <ValueTypeFQN> version <N>
  -- DO NOT EDIT — re-run the emitter to regenerate.

  DO $$
  BEGIN
      CREATE TYPE <schema>.<type_name> AS (
          <field-list, in DSL declaration order>
      );
  EXCEPTION
      WHEN duplicate_object THEN NULL;
  END $$;
  ```
  The `DO $$ ... EXCEPTION WHEN duplicate_object THEN NULL; END $$` envelope is the idempotent-`CREATE TYPE` idiom — PG lacks `CREATE TYPE IF NOT EXISTS`. Composite-type field column names follow snake_case mapping (FR-451).
- **FR-422 (MUST, I).** Each `EnumDecl` at namespace scope produces `<output>/schema/<namespace-path>/<EnumName>.sql` (or `.v<Version>.sql`) of shape:
  ```
  -- Gravity DSL — generated by gravc (target: postgres-ddl)
  -- source: <EnumFQN> version <N>
  -- DO NOT EDIT — re-run the emitter to regenerate.

  DO $$
  BEGIN
      CREATE TYPE <schema>.<enum_name> AS ENUM (
          '<variant-1>',
          '<variant-2>',
          ...
      );
  EXCEPTION
      WHEN duplicate_object THEN NULL;
  END $$;
  ```
  Variant order is **DSL declaration order** (not sorted), matching FR-354 for JSON Schema enum schemas — variant ordering carries semantic meaning (the first state is the implicit initial state per FR-033) and the round-trip property requires it. Variant names are emitted verbatim (no case change); PostgreSQL enum literals are case-sensitive identifiers.
- **FR-423 (MUST, I).** The entity-table column list (FR-420) is populated in this fixed order:
  1. **Identity column** (one column, named per `EntityDecl.Identity.FieldName` mapped through FR-451; typed per FR-430; `NOT NULL`).
  2. **State column** (one column, named `state`; typed `<schema>.<entity_name>_state` referring to the per-entity lifecycle enum emitted alongside — FR-424; `NOT NULL`).
  3. **Property columns**, in DSL declaration order — one column per `PropertyDecl`, named per FR-451, typed per FR-430, `NOT NULL` unless `T?`.
  4. **Relation columns**, in DSL declaration order — one column per `RelationDecl`. Cardinality-one → `<relation>_id` of the target's identity primitive type (`UUID` in the documented norm). Cardinality-many → `<relation>_ids` of `<identity-type>[]` (Postgres native array). `NOT NULL` unless `IsOptional`; cardinality-many arrays default to `'{}'::<identity-type>[]` so a freshly-inserted row has an empty array rather than `NULL` (cleaner SQL ergonomics; an `IsOptional` cardinality-many drops the default and accepts `NULL`).
  5. **Primary key constraint** referencing the identity column.
- **FR-424 (MUST, I).** Each entity emits an **inline lifecycle enum** alongside its table: `<schema>.<entity_name>_state` (an enum type) is created in the same baseline file as the entity's `CREATE TABLE`, immediately **before** the `CREATE TABLE` statement, wrapped in the same `DO $$ ... EXCEPTION WHEN duplicate_object THEN NULL; END $$` envelope as FR-422. Variant list is the entity's lifecycle states in DSL declaration order. The state-enum type is **entity-local** by name (`<entity>_state`) so two entities with identical state-set names (`Active`, `Inactive`) don't collide. Pre-flight: if a user property is named `state` (would collide with the reserved state column), the emitter emits `PG002` (FR-470) and skips the entity.
- **FR-425 (MUST, I, IV).** Multi-version coexistence: when `model.Declarations` contains two or more `EntityDecl` entries with the same FQN at different versions, **every** schema file for that FQN carries the `.v<Version>` infix. The baseline schema for each version reflects the **state of that version's columns** — `Employee.v1.sql` contains exactly the columns declared in `Employee@1`, `Employee.v2.sql` contains those plus the additive columns from `Employee@2`. The state enum and table name in the multi-version case are likewise version-suffixed: `<entity_name>_v<N>_state`, `<entity_name>_v<N>`. The single-version case keeps the unqualified names (`employee`, `employee_state`).
- **FR-426 (MUST, IV).** **Per-version migration files.** For each entity FQN, the emitter walks all in-scope versions in ascending order and emits one migration file per version:
  - **V1 (the lowest version in scope)** is the **full baseline migration** — identical content to FR-420's schema file body, but headered as a migration and located at `<output>/migrations/<namespace-path>/<prefix>1__<EntityName>.sql`.
  - **V2, V3, ...** are **additive migration files** — `<output>/migrations/<namespace-path>/<prefix><N>__<EntityName>.sql` — carrying only the **diff** between `V(N-1)` and `V(N)`: `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for each new property and each new relation column; `CREATE INDEX IF NOT EXISTS` for each new index; `ALTER TYPE ... ADD VALUE IF NOT EXISTS` for each new lifecycle state (PG 9.6+ syntax). No `DROP`, no `ALTER COLUMN TYPE`, no narrowing — Principle IV guarantees the DSL never asks for any of these.
  - **The migration prefix** (`V` by default, configurable via `migration_prefix` — LD-20) is prepended to the version number; Flyway naming convention is `<prefix><version>__<description>.sql` and the emitter defaults to it.
- **FR-427 (MUST, II).** Value-type and enum migrations: per-version migration files for `ValueTypeDecl` and `EnumDecl` are emitted **only** when the declaration appears at more than one version in the model. Single-version value types / enums get a baseline `schema/` file but no `migrations/` entry — the baseline file is the migration. Multi-version value types: `migrations/<ns>/<prefix><N>__<TypeName>.sql` carrying `ALTER TYPE ... ADD ATTRIBUTE IF NOT EXISTS` for each new field (PG 9.1+ syntax for composite types). Multi-version enums: `ALTER TYPE ... ADD VALUE IF NOT EXISTS` for each new variant.
- **FR-428 (MUST, I).** A **manifest file** is **not** emitted in v1. (Flyway, sqitch, and golang-migrate each discover migrations by filename convention; an explicit manifest would couple the emitter to one tool's runtime. The naming convention in FR-426 is intentionally Flyway-compatible by default and re-usable by sqitch with the `migration_prefix` override.) A future `--migration-tool flyway|sqitch` config knob is sketched in NG-13 deferrals; v1 ships the filename convention only.

### 4.3 Type mapping

- **FR-430 (MUST, II).** DSL primitive → PostgreSQL column type is a closed-form table. The mapping is implemented as a pure function `PrimitiveKind → string` and is byte-stable. The table:

  | DSL primitive | PG column type | Notes |
  |---|---|---|
  | `String` | `TEXT` | PG's `TEXT` carries no length cap and is the canonical PG idiom; `VARCHAR(n)` is reserved for the `@postgres(max_length: n)` annotation case (FR-441 reserved). |
  | `Int` | `INTEGER` | int32 (-2^31..+2^31-1). |
  | `Long` | `BIGINT` | int64. |
  | `Decimal` | `NUMERIC` | Arbitrary precision; user may pin precision via `@postgres(precision: p, scale: s)` annotation (FR-441 reserved — v1 does not consume). |
  | `Boolean` | `BOOLEAN` | |
  | `Date` | `DATE` | |
  | `DateTime` | `TIMESTAMPTZ` | `TIMESTAMP WITH TIME ZONE`; the PG-idiomatic correct choice for a wire-format `DateTime`. Naive `TIMESTAMP` (no tz) is **not** the default — silent tz drift is a foot-gun the spec refuses. A future annotation may opt into `TIMESTAMP` if a use case appears. |
  | `UUID` | `UUID` | PG native UUID type. |

- **FR-431 (MUST, II).** Optionality modifier (`T?`) controls only the `NOT NULL` constraint on the column: present for `T`, absent for `T?`. Otherwise the column type is identical.
- **FR-432 (MUST, II).** Array modifier (`T[]`) wraps the underlying PG column type in PG's native array syntax: `T` → `<pg-type>`, `T[]` → `<pg-type>[]`. Both modifiers may compose: `String[]?` → `TEXT[]` column, nullable; `String?[]` → `TEXT[]` column, NOT NULL (individual array elements cannot be "optional" in PG arrays — an array's elements are always present once the array exists; the inner `?` is dropped with no diagnostic, matching FR-332's documented asymmetry). Array columns of value-type / enum named types use `<composite-or-enum-type>[]` (PG native array of composite / enum is supported).
- **FR-433 (MUST, II).** **Cardinality-one relation.** Encoded as a **foreign-key column**:
  ```
  <relation_name>_id <identity-type> [NOT NULL] REFERENCES <schema>.<target_table>(<target_identity_column>)
  ```
  Column name: snake_case mapping of `RelationDecl.Name` + `"_id"`. Identity-type: the target entity's identity primitive type mapped through FR-430. `NOT NULL` is emitted unless `IsOptional`. A **btree index** is emitted alongside (separate `CREATE INDEX IF NOT EXISTS` statement): `ix_<entity_name>_<relation_name>_id ON <schema>.<entity_name>(<relation_name>_id)`. Btree is the canonical PG choice for equality / range lookups on a single column — the dominant query plan a foreign-key column participates in.
- **FR-434 (MUST, II).** **Cardinality-many relation.** Encoded as a **native PG array column**:
  ```
  <relation_name>_ids <identity-type>[] [NOT NULL] [DEFAULT '{}'::<identity-type>[]]
  ```
  Column name: snake_case mapping of `RelationDecl.Name` + `"_ids"`. Default empty-array clause is emitted only when the column is `NOT NULL` (avoids the `IsOptional` + `DEFAULT` redundancy). No foreign-key constraint is emitted — PG does not natively enforce FKs on array elements (the operator can layer a constraint trigger downstream; the emitter does not). A **GIN index** is emitted alongside: `ix_<entity_name>_<relation_name>_ids ON <schema>.<entity_name> USING GIN (<relation_name>_ids)`. GIN is the canonical PG choice for "does this array contain element X" lookups — the dominant query plan a cardinality-many relation participates in.
- **FR-435 (MUST, I, V).** **NamedTypeRef resolution** in property positions:
  - Reference to a `ValueTypeDecl` → column type is `<schema>.<value_type_name>` (composite). For multi-version coexistence, the resolved type carries the same `.v<N>` suffix as the referent: `<schema>.<value_type_name>_v<N>`. (Composite-type-versioning in PG is real but rare; the emitter is correct under multi-version even if the underlying use case is uncommon.)
  - Reference to an `EnumDecl` → column type is `<schema>.<enum_name>`. Multi-version suffix as above.
  - Reference to an entity (only legal inside `RelationDecl`, never as a `PropertyDecl.Type` per the resolver's guarantee) → never reached in this position; the relation renderer (FR-433 / FR-434) consumes it.

### 4.4 Annotation consumption

- **FR-440 (MUST, VI).** The `@postgres(key: value, ...)` annotation is consumed on `PropertyDecl`, `IdentityDecl`, and `RelationDecl` in v1. The claimed keyword subset is `{column, index, unique, default}`. Each claimed keyword has a value-type contract (FR-441) the emitter enforces; unrecognised keys within `@postgres(...)` produce `PG003`.
- **FR-441 (MUST, VI).** Per-key value-type contracts (the emitter rejects mismatches with `PG003`):
  - `column` — string. Overrides the default snake_case column name derived from the DSL identifier (FR-451). Useful for legacy-schema mapping. The override must itself be a valid PG identifier (matches `[a-z_][a-z0-9_]*`, length ≤ 63 — PG's `NAMEDATALEN`); invalid overrides produce `PG004` (FR-470).
  - `index` — boolean. When `true`, emit a standalone btree `CREATE INDEX IF NOT EXISTS ix_<entity>_<column>` on the column. Layered on top of the FK indexes from FR-433 (a relation with `@postgres(index: true)` is a no-op — the FK index already exists; the emitter de-duplicates by index name).
  - `unique` — boolean. When `true`, append a `UNIQUE` constraint to the column.
  - `default` — string. Emitted verbatim as the column's `DEFAULT` clause (`DEFAULT <value>`). The string is **not** validated — the user is opting into a PG expression (`'gen_random_uuid()'`, `'CURRENT_DATE'`, `'now()'`) and the emitter trusts the value. Determinism is preserved because the user authored the value deliberately (G-8); the emitter never invents a default.
  - **Reserved (not consumed in v1; rejected with `PG003` if used to make the reservation observable rather than silently accepted): `precision`, `scale`, `max_length`, `storage`, `index_method`, `partial_where`.** Future amendments may consume these; reserving them now prevents a community emitter from squatting the namespace.
- **FR-442 (MUST, VI).** Annotations on `EntityDecl`, `ValueTypeDecl`, `EnumDecl`, `EventDecl`, `CommandDecl` are syntactically reserved by FR-053 but Phase 5 does not consume them; an `@postgres(...)` on these positions is **silently ignored** in v1 (consistent with the Phase 4 FR-343 precedent).
- **FR-443 (MUST, VI).** Annotation namespace ownership contract (FR-052 / `HOST002`): a second registered emitter claiming `"postgres"` causes the host's discovery pass to emit `HOST002` and refuse to run either emitter.

### 4.5 Determinism

- **FR-450 (MUST, "Deterministic output").** Keyword case: SQL keywords are **uppercase** (`CREATE TABLE`, `IF NOT EXISTS`, `NOT NULL`, `REFERENCES`, `PRIMARY KEY`, `USING GIN`); identifiers (table names, column names, type names, index names, schema names) are **lowercase**. This matches the canonical PG documentation convention and the `pg_dump` default. Quoted identifiers are **not** emitted in v1 — every name the emitter produces is a valid PG bare identifier (snake_case, starts with letter, ASCII only).
- **FR-451 (MUST, "Deterministic output", V).** Identifier mapping: DSL identifiers are mapped to PG identifiers by a deterministic snake_case pass. Rules:
  - DSL `first_name` (already snake_case) → PG `first_name`.
  - DSL `firstName` (camelCase) → PG `first_name`.
  - DSL `FirstName` (PascalCase) → PG `first_name`.
  - DSL `URL` (all caps) → PG `url`.
  - DSL `HTTPSResponse` (mixed acronyms) → PG `https_response` (every uppercase run becomes one lowercase token; a transition from lowercase to uppercase inserts an underscore; a transition from uppercase to lowercase inserts an underscore *before* the final uppercase letter — the standard "acronym-aware" snake_case algorithm).
  - DSL `Employee` (entity name) → table `employee`.
  - DSL `Lifecycle.States[i]` (enum variant, may be PascalCase) → emitted **verbatim** inside `CREATE TYPE ... AS ENUM` because enum literals are string-equal-matched at the DB layer and changing their case would silently break consumer queries. Variant identifiers in the source are **already** required to match `[A-Z][A-Za-z0-9_]*` by FR-024.
  - User-supplied `@postgres(column: "<name>")` overrides the algorithm verbatim (FR-441).
- **FR-452 (MUST, "Deterministic output").** Column ordering inside each `CREATE TABLE`: identity column → state column → property columns in DSL declaration order → relation columns in DSL declaration order (FR-423). No alphabetical re-sort. Inside each `CREATE TYPE ... AS (...)` composite, fields are emitted in DSL declaration order (FR-421). Inside each enum, variants are emitted in DSL declaration order (FR-422).
- **FR-453 (MUST, "Deterministic output").** Constraint and index ordering: foreign-key constraints emitted **after** the `CREATE TABLE` statement (as separate `ALTER TABLE ... ADD CONSTRAINT` statements rather than inline `REFERENCES` clauses) are sorted **ordinally by constraint name**. Standalone index statements (`CREATE INDEX IF NOT EXISTS`) are sorted **ordinally by index name**. This guarantees that adding a new relation produces a minimal-diff change in the emitted file.
- **FR-454 (MUST, "Deterministic output").** Line endings LF, UTF-8 no BOM. Indentation: 4 spaces (matches the `pg_dump` default and is the convention `EXPLAIN ANALYZE` output uses for nested plan trees). One blank line between top-level statements (`CREATE TABLE` / `DO $$ END $$` / `ALTER TABLE` / `CREATE INDEX`); no trailing whitespace; file ends with a single trailing newline. No `Environment.NewLine`, no `StringBuilder.AppendLine`; the emitter uses an explicit `"\n"` literal everywhere.
- **FR-455 (MUST, "Deterministic output").** Path ordering: `BufferedEmitterOutput` already sorts paths under `StringComparer.Ordinal`; the emitter writes `schema/<ns>/<name>.sql` and `migrations/<ns>/<prefix><N>__<name>.sql` directly through the sink. The migration prefix's ordinal value relative to `schema/` is `m < s`, so when the host commits files to disk in path order, migrations land before schema files — operators expect to apply migrations first, which the sort order naturally produces.

### 4.6 Configuration + packaging

- **FR-460 (MUST, VI).** The `Gravity.Dsl.Emitter.PostgresDdl` NuGet package layout mirrors `Gravity.Dsl.Emitter.JsonSchema`'s exactly:
  - `lib/net9.0/Gravity.Dsl.Emitter.PostgresDdl.dll` — the emitter assembly.
  - `buildTransitive/Gravity.Dsl.Emitter.PostgresDdl.props` — contributes the emitter DLL path to `<GravityDslEmitterAssembly>` so the Phase 9 MSBuild target discovers it.
  
  No `tasks/`, no `tools/`. `<IsPackable>true</IsPackable>`, `<Deterministic>true</Deterministic>`, `<EmbedUntrackedSources>true</EmbedUntrackedSources>`.
- **FR-461 (MUST, "Deterministic output").** Packaging is byte-deterministic — `dotnet pack Gravity.Dsl.Emitter.PostgresDdl -c Release` twice on a clean tree produces byte-identical `.nupkg` files. Asserted by integration test (AC-5.9).
- **FR-462 (MUST, VI).** Banned-APIs analyzer enabled (the emitter falls within the analyzer scope per `Directory.Build.props`). No `DateTime.UtcNow`, no `Environment.MachineName`, no `Path.GetTempFileName`, no `Random`, no `String.ToUpper` / `String.ToLower` without `*Invariant` suffix. Determinism is enforced at the analyzer layer; the emitter codebase has no legitimate need for clock or machine identity.
- **FR-463 (MUST, VI).** Zero third-party `PackageReference`. The emitter depends on the .NET 9 BCL only. No `Npgsql`, no `Dapper`, no SQL-formatting library — string assembly is straightforward and a third-party dependency would add a determinism risk for no benefit.
- **FR-464 (MUST, VI).** Rule ids introduced by Phase 5, all in the `PG` (PostgreSQL DDL emitter) namespace:
  - `PG001` — invalid `schema` config value (not a valid PG identifier per `[a-z_][a-z0-9_]*`, length ≤ 63). Severity `Error`. The emitter aborts before writing any file.
  - `PG002` — entity declares a property named `state` (collides with the reserved lifecycle-state column — FR-424). Severity `Error`. The entity is skipped; emission continues for other declarations.
  - `PG003` — unknown `@postgres` annotation key, OR annotation value type mismatch (FR-441), OR a reserved-but-not-consumed key (`precision`, `scale`, `max_length`, `storage`, `index_method`, `partial_where`). Severity `Error`.
  - `PG004` — `@postgres(column: "<value>")` override is not a valid PG identifier. Severity `Error`.
  - `PG005..PG010` — Reserved for forward use; not consumed by this slice.
- **FR-470 (MUST, V).** Diagnostic format: every `PG*` diagnostic carries a `SourceSpan` pointing at the offending construct in `.gravity` source (the relevant `PropertyDecl.Span`, `AnnotationDecl.Span`, or a synthetic span against the emitter's `TargetName` for config-block diagnostics — analogous to `HOST001` / `HOST002`). Messages name the rule, the entity / value type / enum FQN, the property / annotation key, and (for `PG003` / `PG004`) the expected value type or identifier rule. Sample: `"property 'first-name' on entity 'hr.Employee' produces column name 'first_name' but @postgres(column: '1bad') is not a valid PostgreSQL identifier at <path>:<line>:<col>"`.

- **AC-5.1.** **Build + pack round-trip.** `dotnet build Gravity.Dsl.Emitter.PostgresDdl -c Release` succeeds with zero warnings (treat-warnings-as-errors). `dotnet pack ... -c Release` produces `Gravity.Dsl.Emitter.PostgresDdl.<version>.nupkg` with exactly `lib/net9.0/Gravity.Dsl.Emitter.PostgresDdl.dll`, `buildTransitive/Gravity.Dsl.Emitter.PostgresDdl.props`, and the standard NuGet metadata.
- **AC-5.2.** **Registry registration and ownership.** `EmitterRegistry.FromInstances(new IEmitter[] { new PostgresDdlEmitter() })` produces a registry with `TargetName == "postgres-ddl"`, `AnnotationNamespace == "postgres"`, `SupportedAstVersions.Satisfies("1.0.0") && SupportedAstVersions.Satisfies("1.1.0")`. A second registration alongside a stub emitter that also claims `"postgres"` surfaces `HOST002` naming both claimants and aborts the run. A three-way registration alongside `JsonSchemaEmitter` and `CSharpEmitter` succeeds (the three namespaces `postgres`, `json_schema`, `csharp` are disjoint).
- **AC-5.3.** **End-to-end against `samples/registry/`.** Running the emitter against the HR registry sample produces exactly: `schema/hr/Employee.sql`, `schema/hr/TimeEntry.sql`, `schema/hr/Project.sql`, one `schema/hr/<TypeName>.sql` per namespace-scope value type, one `schema/hr/<EnumName>.sql` per namespace-scope enum, plus `migrations/hr/V1__Employee.sql`, `migrations/hr/V1__TimeEntry.sql`, `migrations/hr/V1__Project.sql`. Zero diagnostics. File count and ordinal-sorted paths byte-checked.
- **AC-5.4.** **Generated DDL is syntactically valid PostgreSQL 14.** A test fixture parses every emitted `.sql` file with a PG-14-compatible parser (the `Antlr4.Runtime` PostgreSQL grammar via `PgQuery.Net` test-only dependency, or — if the dependency footprint is too large — a `psql --dry-run-equivalent` shell-out under the integration-harness umbrella). No syntax errors reported.
- **AC-5.5.** **Idempotency.** A test applies every emitted `schema/` file to a fresh in-process Postgres instance (`Testcontainers.PostgreSql` test-only dependency under the integration harness, or a stubbed equivalent if the runner cannot pull container images — gated by an environment flag `GRAVITY_PG_E2E=1`), then **applies the files a second time**. Both passes succeed; the database state is byte-identical after pass 1 and pass 2 (verified by `pg_dump --schema-only` comparison). This validates the `IF NOT EXISTS` / `DO $$ ... EXCEPTION` envelope.
- **AC-5.6.** **Type-mapping table golden.** A parameterised fixture exercises every row of FR-430: one entity per primitive (single property of that type), plus modifier combinations (`T`, `T?`, `T[]`, `T[]?`, `T?[]`, `T?[]?`). Each emitted file is byte-checked against a golden under `tests/golden/postgres-ddl/primitive-matrix/`. The `Decimal` fixture's golden carries `NUMERIC`; the `DateTime` fixture carries `TIMESTAMPTZ`; the `String[]` fixture carries `TEXT[] NOT NULL DEFAULT '{}'::TEXT[]`; the `String[]?` fixture carries `TEXT[]` with no default.
- **AC-5.7.** **Configurable target schema.** A run with `schema: "hr_prod"` produces tables under `hr_prod.<table>` everywhere — every `CREATE TABLE`, every `REFERENCES`, every `CREATE TYPE` carries the `hr_prod.` qualifier. A run with `schema: "public"` (default) produces the same content with `public.` qualifier. A run with `schema: "1bad"` (invalid PG identifier) produces `PG001` and zero output files. A run with `schema: "tenant_42"` succeeds.
- **AC-5.8.** **Migrations for entity versions.** A fixture with two `EntityDecl` entries for the same FQN at versions 1 and 2 (v2 adds two new properties, one new relation, one new lifecycle state) produces: `schema/hr/Employee.v1.sql`, `schema/hr/Employee.v2.sql`, `migrations/hr/V1__Employee.sql` (baseline), `migrations/hr/V2__Employee.sql` (additive). The V2 migration content carries exactly: two `ALTER TABLE hr.employee_v2 ADD COLUMN IF NOT EXISTS <name> <type>` (one per new property), one `ALTER TABLE hr.employee_v2 ADD COLUMN IF NOT EXISTS <relation>_id UUID NOT NULL REFERENCES ...` plus the matching `CREATE INDEX IF NOT EXISTS`, and one `ALTER TYPE hr.employee_v2_state ADD VALUE IF NOT EXISTS '<NewState>'`. No `DROP`. No `ALTER COLUMN TYPE`. Byte-checked against golden `tests/golden/postgres-ddl/multi-version/`.
- **AC-5.9.** **Byte-deterministic across runs and platforms.** The emitter runs twice in a single process against `samples/registry/` and produces byte-identical buffers. Same suite on Linux + macOS CI runners produces byte-identical files. `dotnet pack Gravity.Dsl.Emitter.PostgresDdl` twice on a clean tree produces byte-identical `.nupkg` files.
- **AC-5.10.** **Relations produce the expected indexes.** A test fixture with one cardinality-one relation (`assigned_to: Employee`) and one cardinality-many relation (`tags: Tag*` — using the DSL's `many` cardinality syntax) on the same entity produces, in the entity's `schema/` file: a `<relation>_id UUID NOT NULL REFERENCES ...` column for the one, a `<relation>_ids UUID[] NOT NULL DEFAULT '{}'::UUID[]` column for the many, a `CREATE INDEX IF NOT EXISTS ix_<entity>_assigned_to_id ON ...(assigned_to_id)` btree, and a `CREATE INDEX IF NOT EXISTS ix_<entity>_tags_ids ON ... USING GIN (tags_ids)` GIN. The two index statements are sorted ordinally by index name in the file (FR-453).
- **AC-5.11.** **`@postgres` annotations fold into output verbatim.** A fixture entity declares `properties { email: String @postgres(column: "primary_email", unique: true, index: true); employee_code: String @postgres(default: "'EMP-' || nextval('seq')"); }` and the emitted `CREATE TABLE` carries: `primary_email TEXT NOT NULL UNIQUE`, `employee_code TEXT NOT NULL DEFAULT 'EMP-' || nextval('seq')`, plus `CREATE INDEX IF NOT EXISTS ix_<entity>_primary_email ON ...(primary_email)`. The byte order of statements (table → indexes) is asserted by golden.
- **AC-5.12.** **Annotation key validation (`PG003` / `PG004`).** Fixtures producing each diagnostic sub-cause: (a) `@postgres(unknown_key: "x")` → `PG003` ("unknown @postgres key"). (d) `@postgres(precision: 10)` → `PG003` (reserved key not consumed in v1). (e) `@postgres(unique: "yes")` → `PG003` (unique must be boolean).
