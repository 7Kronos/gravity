# Gravity DSL — Specification (Phase 4: JSON Schema reference emitter)

**Status:** Locked for implementation
**Date:** 2026-05-18
**Driven by:** `docs/specs.md` §5 (Pluggable emitter architecture), §5.2 (Reference emitters shipped with v1 — JSON Schema row), §6.3 (project layout, `Gravity.Dsl.Emitter.JsonSchema/`), §7 phase 4 ("JSON Schema reference emitter"); constitution Principle VI (Pluggable, not prescriptive — dominant for this phase) and the "Deterministic output" architectural constraint (Principle I gates artifact-as-spec).
**Predecessor:** `specs/003-phase-9-build-integration/spec.md` (Phase 9 — MSBuild target + reference sample outline emitter). The MSBuild target's emitter-discovery, packaging, and parity guarantees from that phase apply unchanged to this emitter; Phase 4 ships **a new reference emitter** behind that already-existing host surface. Phase 8 versioning (`specs/002-phase-8-versioning/spec.md`) is also a hard predecessor — the per-entity `version` integer and `NamedTypeRef.Version` field carried on the AST are observable in the JSON Schema surface (FR-310, FR-318).
**Scope statement:** This is the lockable spec for Phase 4 of `docs/specs.md` §7: a NuGet-distributed JSON Schema reference emitter (`Gravity.Dsl.Emitter.JsonSchema`) that consumes the existing Phase 0–3 / Phase 8 AST and emits per-entity, per-value-type, and per-enum JSON Schema bundle files in **Draft-07**. Phases 5–7 (GraphQL SDL, OpenAPI, AsyncAPI) remain future work; the JSON Schema emitter does not anticipate their needs. Phase 10 (OSS launch) and the proposed `--draft 2020-12` multi-dialect surface remain explicitly deferred.

## 1. Goals (tied to constitution)

| Goal | Constitution principle |
|---|---|
| G-1. Every JSON Schema artifact — per-entity-state, per-event-payload, per-command-request, per-command-response, per-value-type, per-enum — is generated from `.gravity` source. No hand-authored `.json` schemas are admitted into any sample, fixture, or downstream consumer; the emitter is the single source of those bytes. | I. The DSL is the spec. |
| G-2. The emitter declares no storage, transport, or deployment vocabulary in its output. Schemas describe domain shape only; selecting a validator, a gateway, or a transport is the consumer's concern and lives outside this artifact. The annotation namespace `json_schema` is restricted to the keyword subset documented in FR-340 — extensions that would smuggle storage hints (`x-pg-column`, `x-mongo-index`) are out of scope and any future request for such must be refused by amendment. | II. Domain-only. |
| G-3. Generated schemas carry `additionalProperties: false` on every object so downstream code cannot append properties the DSL did not declare. Open-content schemas are explicitly **not** an option in v1; the schema is the contract, not a permissive starting point. | III. Read-only at build time. |
| G-4. Per-entity / per-value-type / per-enum schemas surface the AST's `Version` integer through a Gravity-namespaced vendor extension (FR-310) so consumers can read the version without re-parsing `.gravity` source. The vendor-extension prefix (`x-gravity-`) is used because JSON Schema Draft-07 tolerates unknown top-level keywords. | IV. Additive-only versioning by default. |
| G-5. Generated schemas are valid Draft-07 — every reference, every keyword, every type literal is interpretable by AJV, JsonSchema.Net, Python's `jsonschema`, and any HTTP gateway / message validator that handles the dialect. The grammar bar set in Principle V applies to the emitted artifact too: a domain reviewer reading `Employee.json` should understand it without consulting Gravity-internal documentation. | V. AI-readable (extended to "ecosystem-readable" for the schema target). |
| G-6. The JSON Schema emitter is **one plugin among many**: it is discovered through the existing `EmitterRegistry` scan, it claims `json_schema` exclusively (FR-052 / `HOST002` enforces), it declares a stable `IEmitter` contract version, and it ships as a separate NuGet package (`Gravity.Dsl.Emitter.JsonSchema`) — never bundled into the compiler core nor into `Gravity.Dsl.MsBuild`. The Phase 9 MSBuild target consumes it via the documented two-package layout (one for the host, one per emitter). | VI. Pluggable, not prescriptive. |
| G-7. The emitter runs and emits without any Registry artifact present. No scope, permission, release, or library construct appears in the generated schema set. The `$id` URI strategy (FR-318 — relative `$ref`, no absolute base) is deliberately Registry-agnostic so Registry-published schemas can later attach an absolute base without rewriting the emitter. | VII. Composable upward. |
| G-8. Output is byte-deterministic across runs and platforms: sorted keys, sorted `required` arrays, sorted `definitions` entries, declaration-order `properties` and `enum`, LF newlines, UTF-8 no BOM, no timestamps. The Phase 0–3 AC-6a / AC-6b bar is inherited and re-asserted under AC-4.9. | "Deterministic output" (architectural constraint). |

## 2. Non-goals (Phase 4)

- **NG-1.** **JSON Schema Draft 2020-12 / 2019-09 / Draft-06.** Phase 4 emits Draft-07 exclusively (LD-14). A future `--draft 2020-12` flag is sketched for Phase 10+ and is **not** delivered here; the emitter has no multi-dialect dispatch. Consumers who require 2020-12 must wait or write a third-party emitter against the public AST.
- **NG-2.** **OpenAPI emission.** Phase 6 (`Gravity.Dsl.Emitter.OpenApi`) is a separate emitter and a separate spec. The JSON Schema emitter does not produce an OpenAPI document, does not embed schemas in an OpenAPI envelope, and does not co-emit a `components/schemas` section. The two emitters happen to share a dialect lineage but are independent NuGet packages.
- **NG-3.** **AsyncAPI emission.** Phase 7 (`Gravity.Dsl.Emitter.AsyncApi`) is similarly independent. The per-event payload schemas this emitter produces can be referenced by a future AsyncAPI emitter via `$ref`, but no inter-emitter integration is built in Phase 4.
- **NG-4.** **Absolute `$id` URIs.** Phase 4 uses relative `$ref` only. No `$id` carries a registry URL, a CDN base, or any non-relative identifier. The schemas are filesystem-relocatable; consumers who need absolute URIs run a post-processor or wait for the Registry layer.
- **NG-5.** **Schema versioning beyond the Phase 8 AST version surface.** No `$id` with a version segment in v1 (`#/v1/Employee`, `https://gravity.example/hr/employee/v1`, etc.). Multi-version coexistence is reflected in the file path layout (FR-310) and in the `x-gravity-version` vendor extension; the schema bundle for `Employee@2` lives next to the bundle for `Employee@1` and the consumer chooses which file to validate against.
- **NG-6.** **Auto-generation of TypeScript / Python types from JSON Schema.** Tools like `quicktype`, `json-schema-to-typescript`, and `datamodel-codegen` are downstream user concerns. The emitter outputs JSON Schema; what consumers do with it is out of scope.
- **NG-7.** **Runtime validation library.** Selecting a validator (AJV, JsonSchema.Net, Python `jsonschema`, gateway-native engine, etc.) is the consumer's decision. The emitter does not ship a runtime, does not recommend one in the generated artifact, and does not encode validator-specific extensions.
- **NG-8.** **Schema migration / diff tooling.** No `gravc schema-diff v1 v2` command, no migration path generator, no breaking-change detector at the schema layer. The Phase 8 breaking-change pass (`VAL020..VAL030`) runs at the DSL layer and is the single authoritative diff surface; the emitted schemas inherit its guarantees.
- **NG-9.** **Hand-tuneable output formatting.** No `indent: 4`, no `key_order: explicit`, no `pretty: false` knobs in the configuration schema. Determinism is the only formatting contract (FR-350..FR-355); knobs that compromise byte-stability are refused.
- **NG-10.** **JSON Pointer encoding of name segments.** Names that would require JSON Pointer escaping (`/`, `~`) are not legal Gravity identifiers (FR-004 in Phase 0–3 — ASCII letters, digits, underscore, beginning with a letter), so the emitter does not implement `~0` / `~1` escaping. If a future grammar extension permits richer identifiers, escaping becomes a follow-up requirement; Phase 4 does not anticipate it.
- **NG-11.** **Everything in Phase 0–3 / Phase 8 / Phase 9 NGs continues to apply.** Registry features, LSP / formatter / linter, runtime DSL evaluation, storage backends, broker bindings, AI authoring tooling, and the emitter authoring guide (Phase 9b) all remain out of scope.

## 3. Locked decisions (user-approved)

| Decision | Value | One-line rationale |
|---|---|---|
| LD-14. JSON Schema draft | **Draft-07** — `"$schema": "http://json-schema.org/draft-07/schema#"`. Single dialect; no multi-dialect dispatch in v1. A future `--draft 2020-12` flag is Phase 10+. | Draft-07 is universally supported (AJV, JsonSchema.Net, every gateway / message validator, OpenAPI 3.0). Picking one dialect now keeps the emitter, the tests, and the goldens lean; multi-dialect adds combinatorial cost the v1 audience does not need. |
| LD-15. Schema shape per entity | **ALL of** (a) per-event payload schema (one per declared event), (b) per-command **request** schema (arguments) and per-command **response** schema (`ReturnsType`) — two schemas per command, (c) per-entity-state schema (identity + properties + relation reference IDs), (d) per-value-type schema, (e) per-enum schema. The first three live in the entity's bundle's `definitions`; the entity-state schema is the bundle's root. Value-type and enum schemas declared at namespace scope get their own files (LD-16). | Each shape answers a different validator-side question: events validate payloads at ingress, command requests validate RPC arguments, command responses pin the contract the caller can rely on, entity-state schemas validate persisted snapshots and query results, value types / enums are reused leaves. Skipping any of the five would force consumers to hand-author the missing surface, which Principle I forbids. |
| LD-16. Bundle layout | **One bundle file per entity**: `<output>/<namespace path>/<EntityName>.json`. The entity-state schema is the **root** of the bundle; per-event, per-command-request, and per-command-response schemas live in `definitions` keyed by name (FR-311..FR-313). Per-namespace-scope value type / enum: one file each, `<output>/<namespace path>/<TypeName>.json`. Per-entity-internal value type / enum (none in v1 grammar, but reserved): would live in the entity's `definitions` if it existed. Multi-version coexistence (Phase 8): `Employee@1` → `Employee.json` containing `"x-gravity-version": 1`; if `Employee@2` is also in scope, it gets `Employee.v2.json` (FR-310 — the `.vN` infix appears **only** when more than one version of the FQN is in scope; a lone `Employee@1` keeps the unqualified `Employee.json` filename). | One file per entity is the smallest unit that round-trips meaningfully through a validator front-end; sub-splitting (one file per event, one per command) would explode the file count without improving the validator-side workflow. Hoisting namespace-scope value types and enums to their own files matches the `Gravity.Dsl.Emitter.CSharp` per-decl layout and lets consumers compose schemas at the namespace granularity. The `.vN` suffix scheme avoids overwriting when versions coexist while keeping the simple, unqualified filename in the (common) single-version case. |
| LD-17. Target name + annotation namespace | **TargetName** `"json-schema"` (kebab-case, matches the `docs/specs.md` §5.3 example config block). **AnnotationNamespace** `"json_schema"` (underscore, not hyphen — annotation namespaces are identifiers per FR-050, and identifiers in this grammar are `[A-Za-z_][A-Za-z0-9_]*`). The namespace claims the keyword subset `{format, pattern, description, examples, minLength, maxLength, minimum, maximum, multipleOf}` (FR-340). | Target names and annotation namespaces are not the same lexical class and are not required to match — the existing C# emitter has `TargetName = "csharp"` and `AnnotationNamespace = "csharp"` only because both pass both grammars; the JSON Schema emitter cannot, because hyphens are forbidden in annotation namespaces. The split is explicit here so future emitters with hyphenated target names (`community/typescript-records`, etc.) have a precedent. |

## 4. Functional requirements

Levels: **MUST**, **SHOULD**, **MAY**. Each requirement traces to a constitution principle (Roman numeral) or to a named architectural constraint. Phase 4 numbering starts at FR-300 to keep cross-references unambiguous against Phase 0–3 (FR-001..FR-098), Phase 8 (FR-100..FR-161), and Phase 9 (FR-200..FR-260). New rule ids introduced by this phase live in the `JS001..JS010` range for emitter-level diagnostics.

### 4.1 Project + IEmitter contract

- **FR-300 (MUST, VI).** A new project `Gravity.Dsl.Emitter.JsonSchema/Gravity.Dsl.Emitter.JsonSchema.csproj` is introduced as a sibling to `Gravity.Dsl.Emitter.CSharp/`. It targets `net9.0`, inherits `Directory.Build.props` repo-wide settings (deterministic build, banned-APIs analyzer scope, treat-warnings-as-errors), depends on `Gravity.Dsl.Ast` (via `Gravity.Dsl.Ast.csproj` ProjectReference) and `Gravity.Dsl.Emitter` (likewise), and is packed as a stand-alone NuGet package `Gravity.Dsl.Emitter.JsonSchema`. The package is **not** bundled into `Gravity.Dsl.MsBuild` (per Phase 9 FR-210); consumers wanting JSON Schema output add a second `<PackageReference>` analogous to the outline-emitter pattern from LD-12.
- **FR-301 (MUST, VI).** The emitter type `Gravity.Dsl.Emitter.JsonSchema.JsonSchemaEmitter : IEmitter` declares: `TargetName = "json-schema"`, `AnnotationNamespace = "json_schema"`, `SupportedAstVersions = SemanticVersionRange.Parse(">=1.0.0 <2.0.0")`. The version range deliberately spans both the Phase 0–3 `1.0.0` AST and the Phase 8 `1.1.0` AST so the emitter compiles against either without a rebuild; `NamedTypeRef.Version` (Phase 8) is consumed when present and ignored when `null` (FR-318).
- **FR-302 (MUST, VI).** `ConfigurationSchema` declares exactly two keys: `output` (`ConfigValueKind.String`, `Required: true`, no default) and `bundle_strategy` (`ConfigValueKind.String`, `Required: false`, `Default: "per-entity"`). The `bundle_strategy` enum has exactly one legal value in v1 (`"per-entity"`); any other value, including `"per-namespace"`, `"single-file"`, or empty string, is rejected by the emitter's pre-flight with `JS002` (FR-360). Adding a second strategy requires a spec amendment.
- **FR-303 (MUST, VI).** `IEmitter.Emit(model, config, sink)` walks `model.Declarations` in `(FQN ordinal, Version ascending)` order — the FR-161 contract pinned by `ResolvedModel.Declarations` being an `ImmutableSortedDictionary` — and dispatches to per-decl renderers: `EntityDecl` → entity-bundle renderer (FR-310..FR-319), `ValueTypeDecl` → per-value-type-file renderer (FR-316), `EnumDecl` → per-enum-file renderer (FR-317). The renderers produce `JsonElement` / `JsonObject` trees that the emitter serialises through `System.Text.Json` with a custom sorted-key writer (FR-350); each schema is then handed to `IEmitterOutput.WriteFile` with a `/`-normalised relative path.
- **FR-304 (MUST, VI).** Configuration ingestion uses the existing `Gravity.Dsl.Emitter.ConfigLoader` (Phase 0–3 FR-081) — the JSON Schema emitter publishes its `EmitterConfigSchema` and the host validates the user's `.gravity.config` block before invocation. The emitter itself does no YAML parsing; it consumes the already-typed `EmitterConfig.Values` dictionary. This mirrors `CSharpEmitter` (lines 142-152) and `OutlineEmitter` (`OutlineEmitterConfig.From`) so a community author copy-pasting this template recognises the idiom.
- **FR-305 (MUST, VI).** The emitter is `sealed` and has a public parameterless constructor — the `EmitterRegistry` discovery contract from Phase 0–3 (`AppendEmittersFromAssembly`, line 96) requires both. No `partial`, no `virtual` members. Renderer helper types are `internal sealed` to keep the public surface area minimal and prevent third-party consumers from extending the emitter through subclassing (Principle III's read-only-at-build-time principle, extended one layer up).

### 4.2 Schema generation

- **FR-310 (MUST, I, IV).** Each `EntityDecl` produces a bundle file at `<output>/<namespace-path>/<EntityName>.json` (single-version case) or `<output>/<namespace-path>/<EntityName>.v<Version>.json` (multi-version case — when `model.Declarations` contains two or more entries with the same `FQN` and different `Version` values, **every** bundle file for that FQN carries the `.v<Version>` infix, including the one for `Version == 1`, so the filename layout is symmetric rather than off-by-one). The namespace path is the DSL namespace with `.` replaced by `/` (matching `NamespaceMapper.ComposeDirectory` from Phase 0–3). The root JSON document has shape:
  ```
  {
    "$schema": "http://json-schema.org/draft-07/schema#",
    "title": "<EntityFQN>",
    "x-gravity-version": <int>,
    "type": "object",
    "properties": { ... entity-state properties (FR-314) },
    "required": [ ... ],
    "additionalProperties": false,
    "definitions": { ... per-event, per-command-request, per-command-response, per-state-enum, per-entity-local-type schemas }
  }
  ```
  The root document **is** the entity-state schema; per-event and per-command schemas are reached exclusively through `definitions` (FR-311..FR-313). The `x-gravity-version` key is the Gravity vendor extension carrying the AST `EntityDecl.Version` integer; Draft-07 tolerates unknown top-level keywords by spec, so consumer validators ignore it without error.
- **FR-311 (MUST, I).** Each `EventDecl` produces a `definitions` entry keyed by the event's `Name`. The entry shape is `{ "title": "<EntityFQN>.<EventName>", "type": "object", "properties": { ... payload fields }, "required": [ ... non-optional payload field names sorted ordinally ], "additionalProperties": false }`. An empty payload (`EventName {};`) produces `{ "title": "...", "type": "object", "properties": {}, "required": [], "additionalProperties": false }`. The `definitions` key is the bare event name (e.g. `Submitted`), **not** an `Events.Submitted` qualified path, because the entity-bundle scope already disambiguates and over-qualification would bloat `$ref`s for no reader benefit.
- **FR-312 (MUST, I).** Each `CommandDecl` produces a `definitions` entry keyed `<CommandName>Request` of shape `{ "title": "<EntityFQN>.<CommandName>.Request", "type": "object", "properties": { ... arguments }, "required": [ ... non-optional argument names sorted ordinally ], "additionalProperties": false }`. Commands with zero arguments produce `{ "title": "...", "type": "object", "properties": {}, "required": [], "additionalProperties": false }`. Argument property entries are emitted in DSL declaration order under `properties` (FR-353); the `required` array is sorted ordinally.
- **FR-313 (MUST, I).** Each `CommandDecl` also produces a `definitions` entry keyed `<CommandName>Response`. The `ReturnsType` is a bare identifier (a value-type or enum name; Phase 8 FR-100 forbids `@N` suffix on `returns`). The response entry is **always** a `$ref` indirection: `{ "$ref": "#/definitions/<ReturnsType>" }` when the response type is entity-local (currently never the case in v1 grammar but reserved); `{ "$ref": "<ReturnsType>.json" }` when the response type is a namespace-scope value type or enum (the common case — Employee's `OnboardResult`, TimeEntry's `ApprovalResult`, etc.). The indirection guarantees that response types appear in exactly one place per file regardless of how many commands return them, and that switching a response type from "local" to "namespace-scope" is a single-line filename change rather than a deep restructuring.
- **FR-314 (MUST, I, II).** The entity-state schema's `properties` are populated, in DSL declaration order, by:
  1. The identity field (one entry, keyed by `EntityDecl.Identity.FieldName`, typed per FR-330).
  2. Every property in `EntityDecl.Properties` (typed per FR-330, annotations folded in per FR-340).
  3. Every relation in `EntityDecl.Relations` — relations are encoded as identity references, **not** as nested objects. Cardinality `one` produces `{ "type": "string", "format": "uuid" }` (when the target entity's identity is `UUID`, which is the documented norm); cardinality `many` produces `{ "type": "array", "items": { "type": "string", "format": "uuid" }, "uniqueItems": true }`. The relation property name is `RelationDecl.Name + "_id"` for cardinality-one and `RelationDecl.Name + "_ids"` for cardinality-many; the `_id` / `_ids` suffix is the convention because the property carries an identifier, not the entity itself, and naming it bare (e.g. `employee`) would invite consumers to embed the full entity object — which the schema does not permit. The relation's `Semantic` clause is recorded under the property entry's `description` field (`"references <Target> by id (<semantic>)"`) so domain reviewers see the role; if no semantic is declared, the description is `"references <Target> by id"`.
- **FR-315 (MUST, I).** The lifecycle state set produces a `definitions` entry keyed `<EntityName>State` of shape `{ "type": "string", "enum": [ ... states in DSL declaration order ] }`. The entity-state schema's `properties` map includes a `state` entry whose body is `{ "$ref": "#/definitions/<EntityName>State" }`. The `state` property is added to the entity-state `required` array. The DSL grammar guarantees at least one state per entity (FR-024 — the `states { ... }` block holds a comma-separated list of state identifiers); a hypothetical zero-state entity would produce `{ "type": "string", "enum": [] }`, which Draft-07 accepts as a valid (unsatisfiable) schema. The `state` property name is reserved by this emitter on the entity-state schema; user property declarations colliding with `state` produce `JS003` (FR-360 reservation list).
- **FR-316 (MUST, I).** Each `ValueTypeDecl` at namespace scope produces a file `<output>/<namespace-path>/<TypeName>.json` (or `<TypeName>.v<Version>.json` under the same multi-version rule as FR-310) of shape `{ "$schema": "http://json-schema.org/draft-07/schema#", "title": "<TypeFQN>", "x-gravity-version": <int>, "type": "object", "properties": { ... fields }, "required": [ ... non-optional field names sorted ordinally ], "additionalProperties": false }`. Field property entries appear in DSL declaration order under `properties`. The file is **stand-alone** — it does not embed its dependencies; cross-references to other value types or enums use `$ref` to a sibling file (FR-318).
- **FR-317 (MUST, I).** Each `EnumDecl` at namespace scope produces a file `<output>/<namespace-path>/<EnumName>.json` (or `<EnumName>.v<Version>.json` per the multi-version rule) of shape `{ "$schema": "http://json-schema.org/draft-07/schema#", "title": "<EnumFQN>", "x-gravity-version": <int>, "type": "string", "enum": [ ... variants in DSL declaration order ] }`. Enum variant order is **not** sorted; it follows DSL declaration order so that emitter output stays aligned with the source's semantic ordering (e.g. `ContactMethod { Email, Phone, None }` produces `"enum": ["Email", "Phone", "None"]`, not the ordinally-sorted `["Email", "None", "Phone"]`).
- **FR-318 (MUST, I, IV).** `$ref` strategy is two-tier and exclusive: **within-bundle** refs (a property's type is an entity-local definition in `definitions` — currently the entity-state schema's `state` property, the per-command-response indirection, and any future entity-internal value type / enum) use `"$ref": "#/definitions/<Name>"`. **Cross-file** refs (a property's type is a namespace-scope `ValueTypeDecl` or `EnumDecl` declared in the same DSL namespace, or in an imported namespace) use `"$ref": "<RelativeName>.json"` for same-namespace siblings, or `"$ref": "../<other-namespace-path>/<Name>.json"` for cross-namespace references resolved through the same dotted-namespace-to-directory mapping as FR-310. **No absolute URIs**. **No `$id`** declared on any schema in v1. Phase 8's `Foo@N` syntax: when `NamedTypeRef.Version` is non-null and multi-version coexistence applies (FR-310 multi-version case is active for that FQN), the ref target is the `.v<N>.json` variant; when `Version` is null, the ref target is the unqualified `<Name>.json` (matching the resolver's "max declared version" rule from Phase 8 FR-126).
- **FR-319 (MUST, III).** Every object schema (root entity-state, every `definitions` entry of `"type": "object"`, every value-type file) carries `"additionalProperties": false`. This is non-negotiable: the schema is the read-only contract, and an open-content schema would let downstream code introduce properties the DSL never declared. Enum schemas (`"type": "string", "enum": [...]`) inherit a tighter equivalent (no `additionalProperties` keyword on string schemas; the `enum` constraint already closes the value space). Array schemas (`"type": "array", "items": {...}`) likewise need no `additionalProperties`.

### 4.3 Type mapping

- **FR-330 (MUST, II).** DSL primitive → JSON Schema fragment is a closed-form table. The mapping is implemented as a pure function `PrimitiveKind → JsonObject` and is byte-stable across runs. The table:

  | DSL primitive | JSON Schema fragment | Notes |
  |---|---|---|
  | `String` | `{ "type": "string" }` | Plain UTF-8 string; no implicit length bound. |
  | `Int` | `{ "type": "integer", "minimum": -2147483648, "maximum": 2147483647 }` | int32 range pinned in the schema so consumers see the bound without re-parsing `.gravity` source. |
  | `Long` | `{ "type": "integer", "minimum": -9223372036854775808, "maximum": 9223372036854775807 }` | int64 range. **Note:** values outside the JavaScript safe-integer range (±2^53) lose precision in JSON parsers that decode `integer` as IEEE-754 double; this is a known JSON property and the schema does not paper over it. AJV and JsonSchema.Net handle the full range correctly because they parse integers as their native 64-bit type. |
  | `Decimal` | `{ "type": "string", "format": "decimal" }` | **String-encoded.** JSON's `number` is IEEE-754 double, which cannot losslessly represent decimal-typed financial / regulatory values. Encoding decimals as strings lets the schema preserve precision; `format: "decimal"` is an emitter-claimed vendor format (no standard Draft-07 `format` keyword for decimals exists) and is reserved by `json_schema` (FR-340 lists it among the claimed keywords). |
  | `Boolean` | `{ "type": "boolean" }` | |
  | `Date` | `{ "type": "string", "format": "date" }` | Draft-07 `format: "date"` is RFC 3339 full-date (`YYYY-MM-DD`). |
  | `DateTime` | `{ "type": "string", "format": "date-time" }` | Draft-07 `format: "date-time"` is RFC 3339 date-time. |
  | `UUID` | `{ "type": "string", "format": "uuid" }` | Draft-07 `format: "uuid"` (added in Draft-07; widely supported). |

- **FR-331 (MUST, II).** Optionality modifier (`T?`) is **not** encoded inside the type fragment; it controls whether the field name appears in the enclosing object's `required` array. A property `email: String?;` produces `properties: { "email": { "type": "string" } }` and the property name `"email"` is **omitted** from `required`. The JSON Schema field is still present (typed) when the consumer supplies it; it is just legal to omit. This is the canonical Draft-07 idiom and matches how AJV / JsonSchema.Net / Python `jsonschema` interpret optional fields. The emitter does **not** emit `"type": ["string", "null"]` for optionals; nullable-via-union is a Draft 2019-09+ idiom and would conflict with the Draft-07 dialect lock (LD-14).
- **FR-332 (MUST, II).** Array modifier (`T[]`) wraps the underlying type in `{ "type": "array", "items": <T-fragment> }`. Both modifiers may apply: `String[]?` → optional array, encoded as the array fragment present in `properties` but the property name absent from `required`; `String?[]` → array of optional strings, encoded as `{ "type": "array", "items": { "type": "string" } }` because individual array elements cannot be "optional" in JSON Schema (an element is either present in the array or it isn't — JSON arrays don't have "absent" slots). When the inner-type `?` is observed, the emitter **drops** it on the array-items fragment with no diagnostic; this is the documented JSON Schema modelling of `T?[]` and the alternative (`{ "anyOf": [ <T>, { "type": "null" } ] }`) would push the schema toward Draft 2019-09 idioms. **Document this asymmetry in the AC**: `String?[]` and `String[]` produce identical `items` fragments, the only difference being whether the surrounding property name appears in `required`.
- **FR-333 (MUST, II).** `NamedTypeRef` resolution depends on what the resolver attached:
  - Reference to an entity (used only in relations per FR-022; the resolver guarantees this) → see FR-314; the relation property carries `{ "type": "string", "format": "uuid" }` (cardinality-one) or `{ "type": "array", "items": { "type": "string", "format": "uuid" }, "uniqueItems": true }` (cardinality-many), and the `NamedTypeRef` itself does not appear in `properties` directly — it is consumed by the relation renderer.
  - Reference to a `ValueTypeDecl` (anywhere a `TypeRef` is legal — property type, event payload field, command argument, command response, identity type — though identity-as-value-type is an edge case the validator may warn on) → `$ref` per FR-318.
  - Reference to an `EnumDecl` → `$ref` per FR-318.
  - `IsOptional` and `IsArray` modifiers compose as for primitives (FR-331 / FR-332). An array of `$ref` types is `{ "type": "array", "items": { "$ref": "<Name>.json" } }`.
- **FR-334 (MUST, II).** Identity-field type (FR-021) maps as a regular primitive per FR-330. Phase 0–3 documents `UUID` as the canonical identity primitive but accepts any primitive; the emitter does not narrow. Non-`UUID` identity primitives flow through verbatim — e.g. `identity code: String;` produces `properties: { "code": { "type": "string" } }` in the entity-state schema.

### 4.4 Annotation consumption

- **FR-340 (MUST, VI).** The `@json_schema(key: value, ...)` annotation is consumed on `PropertyDecl` only. The claimed keyword subset is `{format, pattern, description, examples, minLength, maxLength, minimum, maximum, multipleOf}`. Each claimed keyword has a **value-type contract** the emitter enforces (FR-341); unrecognised keys within `@json_schema(...)` produce `JS001`. Claimed-keyword values fold directly into the property's schema fragment, replacing the slot the keyword names. Example: `email: String @json_schema(format: "email", pattern: "^.+@.+$", description: "Primary contact email");` produces:
  ```
  "email": {
    "type": "string",
    "format": "email",
    "pattern": "^.+@.+$",
    "description": "Primary contact email"
  }
  ```
  Annotation keys appear in the output **sorted ordinally** under the property entry (FR-352), regardless of the order they appeared in the source. The DSL grammar's `ImmutableSortedDictionary<string, AnnotationValue>` on `AnnotationDecl.Arguments` already enforces this at the AST layer; the emitter re-asserts the ordering at write time as defence-in-depth.
- **FR-341 (MUST, VI).** Per-key value-type contracts (the emitter rejects mismatches with `JS001`):
  - `format` — string. Recognised values (passed through verbatim): `"email"`, `"uri"`, `"uuid"`, `"date"`, `"date-time"`, `"time"`, `"hostname"`, `"ipv4"`, `"ipv6"`, `"regex"`, `"decimal"` (the emitter's own vendor format). Unrecognised values are passed through with a SHOULD-warning `JS004` so consumers using newer Draft-07-compatible formats (e.g. `"duration"` from later drafts) are not blocked, but get a heads-up that the format is not in the emitter's known set.
  - `pattern` — string (interpreted as ECMA-262 regex per Draft-07; the emitter does not compile or validate the regex itself).
  - `description` — string.
  - `examples` — string OR array of strings (in Phase 4; Draft-07's `examples` is an array of arbitrary values, but the annotation grammar's value kinds (FR-050) are scalar so an array literal is not legal — the emitter wraps a single-string value into a one-element array on emission).
  - `minLength` / `maxLength` — integer ≥ 0.
  - `minimum` / `maximum` — integer or decimal.
  - `multipleOf` — integer or decimal ≥ 0 (Draft-07 also forbids zero; the emitter rejects `multipleOf: 0` with `JS001`).
- **FR-342 (MUST, VI).** Annotation keys whose namespace is `json_schema` but key is **not** in the claimed set produce `JS001` ("unknown json_schema key '<key>'; the json_schema namespace claims {format, pattern, description, examples, minLength, maxLength, minimum, maximum, multipleOf}"). This is **emitter-side validation**, not validator-side: Phase 0–3 `VAL006` only fires for *unclaimed namespaces*, and `json_schema` is a claimed namespace, so the validator's check passes. The emitter has the contextual knowledge to enforce the per-key contract because only it knows which keys are meaningful in its output dialect. `JS001` is emitted with severity `Error` (it indicates a misuse the compiler cannot guess past).
- **FR-343 (MUST, VI).** Annotations on positions other than `PropertyDecl` (e.g. `@json_schema` on a `RelationDecl`, an `EventDecl`, a `CommandDecl`, a `ValueTypeDecl`, an `EnumDecl`) are syntactically reserved by FR-053 from Phase 0–3 but Phase 4 does not consume them; an `@json_schema(...)` on a non-property position is **silently ignored** in v1. This is consistent with the Phase 0–3 behaviour for the `@csharp` namespace and avoids over-promising surface that Phase 4 does not exercise. A future amendment may fold `@json_schema(description: "…")` on entities / value types / enums into the corresponding `description` keyword on the bundle's root schema; the AC bar deliberately does not cover this yet.
- **FR-344 (MUST, VI).** The annotation namespace ownership contract (Phase 0–3 FR-052 / `HOST002`) applies: if a second registered emitter were to also claim `"json_schema"`, the emitter host's discovery pass (`EmitterRegistry.Build`) emits `HOST002` naming both claimants and refuses to run either. AC-4.10 pins this through the JSON Schema emitter's registration.

### 4.5 Determinism

- **FR-350 (MUST, "Deterministic output", VI).** Output is serialised through `System.Text.Json.Utf8JsonWriter` with `JsonWriterOptions { Indented = true, IndentCharacter = ' ', IndentSize = 2, NewLine = "\n" }` (or the project's pinned equivalent on .NET 9 where the property names differ — the emitter normalises). UTF-8 encoding, no BOM. The writer is wrapped in a custom `SortedKeyJsonWriter` that buffers each object's key-value pairs and emits them in the order dictated by FR-351..FR-355 below.
- **FR-351 (MUST, "Deterministic output").** Top-level keys of every JSON document (root schemas in entity bundles, value-type files, enum files) appear in the **fixed canonical order**: `$schema`, `title`, `x-gravity-version`, `type`, `properties`, `required`, `additionalProperties`, `enum`, `definitions`. Keys not in the canonical list (e.g. annotation-folded `description`, `format`, `pattern`, `minLength`, `maxLength`, `minimum`, `maximum`, `multipleOf`, `examples`) appear after the canonical list, sorted ordinally. This is the order a human reader expects to encounter Draft-07 keywords and matches the conventional ordering used by `ajv-cli`, JsonSchema.Net's pretty-printer, and the Draft-07 metaschema itself.
- **FR-352 (MUST, "Deterministic output").** Within every object schema's `properties` map, **entries appear in DSL declaration order** (NOT sorted). This matches the existing C# emitter convention (Phase 0–3 FR-094 produces records with positional members in declaration order) and preserves the semantic ordering authors expressed in `.gravity`. Inside each property entry, the schema fragment's keys follow FR-351 (canonical order then sorted overflow). Annotation-folded keys (FR-340) sort ordinally within the overflow section.
- **FR-353 (MUST, "Deterministic output").** `required` arrays are sorted **ordinally**, not in DSL declaration order. This is the documented Draft-07 convention (`required` is a set, and serialising sets in declaration order leaks ordering semantics where none exist) and matches how AJV's schema-pretty-printer and JsonSchema.Net produce their output.
- **FR-354 (MUST, "Deterministic output").** `enum` arrays — both DSL-`EnumDecl` variants (FR-317) and `state` enum entries (FR-315) — appear in **DSL declaration order**, not sorted. This is the C# emitter's convention from Phase 0–3 and the AC-1 round-trip property requires it (variant ordering is observably part of the AST). The asymmetry with `required` (sorted) is intentional: enums carry semantic ordering ("the first state is the implicit initial state" per FR-033), required-arrays do not.
- **FR-355 (MUST, "Deterministic output").** `definitions` entries are sorted **ordinally by key**. This is the equivalent of the Phase 0–3 path-sorted commit order in `BufferedEmitterOutput` and ensures that adding a new event or command to an entity produces a minimal-diff change in the bundle file. The sort happens at write time, not at AST-walk time; the AST walk produces a `Dictionary<string, JsonObject>` of `definitions` entries which is then sorted by key before emission.

### 4.6 Configuration + packaging

- **FR-360 (MUST, VI).** The `Gravity.Dsl.Emitter.JsonSchema` NuGet package layout mirrors `Gravity.Dsl.Emitter.CSharp`'s exactly: the package ships the emitter assembly and the necessary metadata for the Phase 9 MSBuild target to discover it via the `<GravityDslEmitterAssembly>` MSBuild item contract from Phase 9 FR-224. The package contains:
  - `lib/net9.0/Gravity.Dsl.Emitter.JsonSchema.dll` — the emitter assembly (standard NuGet runtime-libraries location).
  - `buildTransitive/Gravity.Dsl.Emitter.JsonSchema.props` — contributes the emitter DLL path to `<GravityDslEmitterAssembly>` so the Phase 9 MSBuild target picks it up; mirrors the outline sample's `.props` shape verbatim.
  No `tasks/` tree, no `tools/` tree — the emitter is loaded by `EmitterRegistry.Discover` (Phase 0–3) at host runtime, not as a build task itself. `<DevelopmentDependency>true</DevelopmentDependency>` and `<IncludeBuildOutput>false</IncludeBuildOutput>` apply so the DLL does not leak into consumers' `bin/` outputs as a runtime reference.
- **FR-361 (MUST, "Deterministic output").** Packaging is byte-deterministic: `dotnet pack Gravity.Dsl.Emitter.JsonSchema -c Release` produces a `.nupkg` whose entries are sorted by path under ordinal comparison and whose internal timestamps are zeroed via the SDK's `ContinuousIntegrationBuild` / `Deterministic` properties (already enabled repo-wide by `Directory.Build.props` lines 11-12). Running `dotnet pack` twice on a clean tree produces byte-identical `.nupkg` files; this is asserted by an integration test (AC-4.9 covers package determinism alongside artifact determinism).
- **FR-362 (MUST, VI).** Banned-APIs analyzer enabled (the emitter falls within the analyzer scope per `Directory.Build.props`). `DateTime.UtcNow`, `Environment.MachineName`, `Path.GetTempFileName()`, and the other banned non-deterministic surfaces remain inaccessible to the emitter codebase. No exception is sought for the emitter — JSON Schema generation has no legitimate need for clock or machine identity.
- **FR-363 (MUST, VI).** Dependency on `System.Text.Json` only — already part of the .NET 9 BCL, no additional NuGet `PackageReference` needed. The emitter does **not** depend on `Newtonsoft.Json`, `JsonSchema.Net`, `Manatee.Json`, or any other third-party JSON library. (The validator test fixture under `tests/integration/json-schema-validator/` may reference `JsonSchema.Net` to validate emitted artifacts against the Draft-07 metaschema (AC-4.4), but that dependency lives in the test project, not the emitter.)
- **FR-364 (MUST, VI).** Rule ids introduced by Phase 4, all in the `JS` (JSON Schema emitter) namespace:
  - `JS001` — unknown `json_schema` annotation key, OR annotation value type mismatch (FR-341), OR `multipleOf: 0`. Severity `Error`.
  - `JS002` — `bundle_strategy` set to a value other than `"per-entity"`. Severity `Error`.
  - `JS003` — user-declared property name collides with the reserved `state` property on the entity-state schema (FR-315). Severity `Error`.
  - `JS004` — `@json_schema(format: "<value>")` carries a format string not in the emitter's known set (FR-341). Severity `Warning` (consumer may be using a newer-draft format string).
  - `JS005..JS010` — Reserved for forward use; not consumed by this slice.
- **FR-365 (MUST, V).** Diagnostic format: every `JS*` diagnostic carries a `SourceSpan` pointing at the offending construct's location in `.gravity` source (the `AnnotationDecl.Span`, `PropertyDecl.Span`, or — for configuration-block diagnostics — the emitter's `TargetName` synthetic span analogous to how `HOST001` / `HOST002` synthesise spans against the target name). Messages name the rule, the entity / value type / enum FQN, the property / annotation key in question, and (for `JS001`) the expected value type. Sample message: `"property 'email' annotation @json_schema(format) expects a string value, got integer literal '42' at <path>:<line>:<col>"`.
