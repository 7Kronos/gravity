# Gravity DSL — Specification (Phase 9c: CI-script test architecture for the MSBuild integration suite)

**Status:** Locked for implementation
**Date:** 2026-05-19
**Driven by:** Phase 9b commit `6c6b80f` carry-over (six `[Fact(Skip=...)]` attributes in `Gravity.Dsl.Tests/MsBuild/` documenting that "these per-AC integration assertions need a separate CI-script test architecture rather than in-process dotnet pack invocation"); `specs/003-phase-9-build-integration/plan.md` §6 risk register row "Smoke test brittleness"; constitution architectural constraint "Build integration parity" (the parity claim is only credible if the parity-pinning tests actually execute on every PR); constitution Quality-standards "Integration tests across the full reference emitter set, run on every PR".
**Predecessor:** `specs/003-phase-9-build-integration/spec.md` (Phase 9 — MSBuild target + reference sample emitter). The acceptance criteria carried into this phase (`AC-9.7-pack`, `AC-9.11..AC-9.15`) are defined there verbatim and are **referenced by id**, not renumbered. `specs/004-phase-4-json-schema-emitter/spec.md` is concurrent with this phase but unaffected — the JSON Schema emitter's golden-file determinism (AC-4.9) runs on the existing in-process `dotnet test` path and does not require the harness specified here.
**Scope statement:** This is the lockable spec for the Phase 9c carry-over slice: a cross-platform integration-test **harness** that runs pack-then-build flows outside `dotnet test` (the SDK build-server's locked `ref/` assemblies make pack-from-within-test unstable; documented in the Phase 9b commit and in the six skipped `[Fact]` attributes), plus the post-pack normalisation needed to satisfy `AC-9.7-pack`. The harness ships alongside the existing `Gravity.Dsl.Tests` fast lane; it does not replace it. The fast lane keeps the unit / golden / smoke suite that already runs cleanly under `dotnet test`. Out of scope: any change to compiler / AST / emitter / MSBuild task semantics; any new functional surface in the DSL; CLI ergonomics polish (still deferred — Phase 9 NG-2 carries forward); the Phase 4–7 reference emitters; the Registry; the `.github/workflows/ci.yml` workflow file itself (a follow-on, non-blocking task; see FR-3040).

## 2. Non-goals (Phase 9c)

- **NG-1.** **Rewriting `Gravity.Dsl.Tests`.** The fast lane stays. The xUnit-based golden, smoke (`MsBuildSmokeTests`), and pack-content (`PackContentTests`) tests that already pass under `dotnet test` are not touched except to replace the six `[Fact(Skip=...)]` lines with thin harness-subcommand wrappers and to remove the dormant `MsBuildIntegrationFixture` class. The harness wraps the same composition helpers; it does not replace the runner.
- **NG-2.** **Cross-version SDK matrix.** Phase 9c locks the harness against the `global.json`-pinned SDK (`9.0.314`). A future `dotnet --version` matrix that runs the harness against 9.0-latest, 10.0-preview, etc. is a separate proposal; AC-9c.* assertions are made only against the pinned version.
- **NG-3.** **Windows CI lane.** The existing CI matrix is Linux + macOS (Phase 4 AC-4.9 documents this verbatim: "the two matrix lanes — Windows is not part of the CI matrix until further notice"). Phase 9c inherits that matrix and does not add Windows. The harness MAY work on Windows by construction (a pure .NET console project), but no `runs-on: windows-latest` leg is added in this phase.
- **NG-4.** **Re-implementing the MSBuild task.** Phase 9c does not change `GravityDslGenTask`, the `.props` / `.targets` files, the bundled `gravc.dll` layout, or any FR-200..FR-260 surface. If a harness step uncovers a real bug in the task, that bug is a separate fix; the harness's job is to make the bug observable, not to fix it.
- **NG-5.** **Replacing `NuGet pack` outright.** The post-pack normaliser approach (LD-22 below) keeps `dotnet pack` as the producer; it does not switch to direct `NuGet.Packaging.PackCommand` API construction. The latter is the cleaner architecture but the larger investment, and Phase 9c is a polish round, not a re-architecture.
- **NG-6.** **CLI ergonomics polish.** Phase 9 NG-2 (`--watch`, `--verbose`, `--color`, progress reporting) carries forward unchanged. The half-finished `AnsiColor.cs` from the Phase 9b session was rolled back; it stays rolled back.
- **NG-7.** **Telemetry / phone-home from the harness.** No HTTP calls, no metrics export, no opt-in usage data. The harness writes to stdout, to a workspace-local log directory, and (optionally) to a JUnit XML file. Nothing leaves the runner.
- **NG-8.** **Test-fixture relocation.** The smoke fixtures under `tests/integration/msbuild-smoke*/` that Phase 9 plan §5.2 enumerated continue to live where they live. The harness consumes them in place. A future cleanup may collapse them; not Phase 9c.
- **NG-9.** **`.github/workflows/ci.yml` ownership.** Phase 9c does **not** create the CI workflow file. A non-blocking follow-on task (FR-3040) wires `dotnet run --project Gravity.Dsl.IntegrationHarness` into the workflow when (and if) that workflow is added by a separate slice. The harness is fully exercised by developer-local `dotnet run` invocations and by the xUnit wrappers (FR-3004) until then.
- **NG-10.** **Everything in Phase 9 NG-1..NG-10 still applies.** The emitter authoring guide question is closed (shipped in Phase 9b); the rest — Roslyn source generator, watch mode, IDE hooks, per-build telemetry, full deterministic-output hardening beyond what we explicitly add here, error-reporting polish beyond Phase 0–3 / Phase 8, Phases 4–7 (Phase 4 shipped separately as `specs/004-*`; Phases 5–7 remain TBD) — are still out of scope.

## 4. Functional requirements

Levels: **MUST**, **SHOULD**, **MAY**. Each requirement traces to a constitution principle (Roman numeral) or to the named architectural constraint. Phase 9c numbering starts at FR-300 to leave headroom; Phase 4 already uses FR-300..FR-365 in its own document tree, but the namespaces are documented in this document only and do not collide with anything cross-referenced from `Gravity.Dsl.*` source. To avoid any ambiguity, **Phase 9c uses FR-3000..FR-3060** (four-digit) to make the numbering visually distinct from every other phase. Rule ids introduced by Phase 9c live in the `HARN` (harness host) namespace, `HARN001..HARN010` — `HARN` follows the existing `MSB`/`HOST`/`CLI`/`PARSE`/`VAL`/`RES`/`LEX`/`CFG` convention of naming the host that emits the rule, not its consumer.

### 4.1 Harness shape and entry point

- **FR-3000 (MUST, "Build integration parity").** The harness is invoked as `dotnet run --project Gravity.Dsl.IntegrationHarness -- <subcommand> [args]` where `<subcommand>` selects a single AC step (`run-ac-9.7-pack`, `run-ac-9.11`, `run-ac-9.12`, `run-ac-9.13`, `run-ac-9.14`, `run-ac-9.15`) or `run-all` to run every step in declaration order. Common flags: `--config <Debug|Release>` (default `Release`), `--out <dir>` for log artefacts (default `<repo>/artifacts/integration-harness/`), `--filter <pattern>` for selecting a subset when combined with `run-all`. The process exit code is `0` on every-subcommand-passed and non-zero on any failure; the failing subcommand's name appears on stderr.
- **FR-3001 (MUST, "Build integration parity", LD-18).** The C# console project `Gravity.Dsl.IntegrationHarness` targets `net9.0`, has `TreatWarningsAsErrors=true`, `Deterministic=true`, and inherits the `Directory.Build.props` repo-wide settings. It is **not** added to `Gravity.Dsl.sln`. The project is buildable via `dotnet build Gravity.Dsl.IntegrationHarness/Gravity.Dsl.IntegrationHarness.csproj` and runnable via `dotnet run --project ...`; both invocations succeed from a clean working tree without first restoring the solution. The harness uses the `global.json`-pinned SDK (`9.0.314`); on startup it logs the observed `dotnet --version` and emits a **warning (not a failure)** on patch-version drift so cross-runner divergence is visible to the reviewer.
- **FR-3002 (MUST, LD-20).** A shared library project `Gravity.Dsl.IntegrationHarness.Shared` at the repo root holds the helpers currently duplicated across `Gravity.Dsl.Tests/MsBuild/MsBuildSmokeTests.cs` and the soon-to-be-deleted `Gravity.Dsl.Tests/MsBuild/MsBuildIntegrationTests.cs`: the `ProcessRunner` static (concurrent stdout/stderr drain + 5-minute timeout), the `WriteConsumerCsproj` factory, the `NuGetConfigFor(string localFeed)` helper, the `MinimalEmployeeGravity` const, the `MinimalBrokenGravity` const (new — for AC-9.3 reuse if a future spec wants to pull AC-9.3 into the harness), and the `Sha256TreeHasher` (new — used by AC-9.5 in the existing parity test and by the AC-9c.* tree-shape assertions). The project is added to `Gravity.Dsl.sln`. `Gravity.Dsl.Tests` adds a `<ProjectReference>` to it and deletes the in-file duplicates. `Gravity.Dsl.IntegrationHarness` adds a `<ProjectReference>` to it.
- **FR-3003 (MUST, "Build integration parity", G-3).** The fast lane and the harness produce byte-identical SHA-256 hashes for the consumer-csproj that they each generate against the same fixture inputs. The shared `WriteConsumerCsproj` helper is the single source of truth; the harness MUST NOT compose its own template. A unit test under `Gravity.Dsl.IntegrationHarness.Tests` (LD-20 sibling, also outside the solution but built by the harness driver) asserts this against a parameter matrix.
- **FR-3004 (MUST, LD-21).** Each of the six previously-skipped `[Fact(Skip=...)]` attributes in `Gravity.Dsl.Tests/MsBuild/MsBuildIntegrationTests.cs` (lines 142, 202, 247, 290, 340) and `Gravity.Dsl.Tests/MsBuild/DeterministicPackTests.cs` (line 19) is replaced with a thin (~3 LOC) `[Fact]` wrapper that invokes `dotnet run --project Gravity.Dsl.IntegrationHarness -- run-ac-<N>` via the shared `ProcessRunner` and asserts: (a) exit code 0, (b) stdout contains the literal substring `AC-<id> PASS`. The `MsBuildIntegrationFixture` class is deleted in the same change. The wrapper Facts carry `[Trait("Category", "Slow")]` so they participate in the slow lane only, matching the existing `MsBuildSmokeTests` placement.

### 4.2 Harness subcommand contracts (AC-mirroring)

Each FR-3010..FR-3015 below maps 1:1 to a carried-over AC from Phase 9. The acceptance criterion id is the contract; the FR records the harness-level execution shape (working directory, command, expected exit code, expected log substring or hash). Diagnostic ids for harness-level failures use the `HARN001..HARN010` range introduced by FR-3050. Each subcommand emits `AC-<id> PASS` to stdout on success (consumed by the FR-3004 wrappers).

- **FR-3010 (MUST, AC-9.11).** Harness subcommand `run-ac-9.11` (`ItemMetadataOverride`) composes a consumer csproj declaring `<PropertyGroup><GravityDslOutputDir>custom-out</GravityDslOutputDir></PropertyGroup>` (a property-level override) against the `MinimalEmployeeGravity` source, runs `dotnet build -c Debug --nologo`, asserts: (a) exit code 0; (b) `custom-out/csharp/Employee.cs` exists and is non-empty; (c) no `Employee.cs` file appears anywhere under `obj/Generated/` of the consumer. NOTE on the mechanism change from Phase 9's original AC-9.11 wording: the Phase 9 spec described `<GravityDsl Include="..."><Output>custom-out/</Output></GravityDsl>` (item metadata override), but MSBuild 17.x reserves the `Output` item-metadata name (raises `MSB4118: The "Output" item metadata name is reserved`). The functional intent of FR-202 / AC-9.11 (route generation to a custom path) is preserved by the `$(GravityDslOutputDir)` property override; per-item override via item metadata is deferred until a non-reserved metadata name is adopted in a future Phase 9 amendment. The harness emits a one-line per-step verdict to stdout and a per-step log file under `--out` capturing the full `dotnet build` stdout+stderr. A failure surfaces `HARN001` ("harness step <name> failed; see <log path>").
- **FR-3011 (MUST, AC-9.12).** Harness subcommand `run-ac-9.12` (`HookOrder`) composes a consumer csproj that includes a `Program.cs` referencing `typeof(hr.Employee)`. The build succeeds only if `GravityDslGenerate` runs before `CoreCompile`. The harness asserts (a) exit code 0; (b) `bin/Debug/net9.0/Consumer.dll` exists; (c) the build log contains a `GravityDslGenerate:` execution marker (the canonical MSBuild target-execution line). The harness extracts the marker via a fixed regex; a missing marker against a successful build is a `HARN002` ("hook order could not be verified — build succeeded but no GravityDslGenerate marker observed; this is likely a verbosity issue but is treated as a real failure to avoid silent regressions").
- **FR-3012 (MUST, AC-9.13).** Harness subcommand `run-ac-9.13` (`EmptyInput`) composes a consumer csproj with no `.gravity` files in its tree and the default `<GravityDsl Include="**/*.gravity" />` glob (which resolves to zero items). The harness asserts: (a) exit code 0; (b) the build log contains zero `PARSE`, `VAL`, `RES`, `LEX`, `HOST`, `MSB`, `JS`, or `CFG` diagnostic-id substrings; (c) no `obj/Generated/` directory exists under the consumer tree. A non-zero exit, a stray diagnostic, or a stray `Generated/` tree surfaces `HARN003`.
- **FR-3013 (MUST, AC-9.14).** Harness subcommand `run-ac-9.14` (`NoGlobalTool`) runs `dotnet tool list --global` in the harness working directory and asserts the resulting stdout does NOT contain the token `gravc` (line-by-line ordinal-comparison). If `gravc` appears, the step surfaces `HARN004` with the remediation hint "uninstall the global tool with `dotnet tool uninstall -g gravc` before re-running the harness". After the assertion passes, the step composes the canonical `MinimalEmployeeGravity` consumer csproj, builds it, asserts exit code 0 and the presence of generated `.cs` under `obj/Generated/csharp/` (or its configured equivalent). This pins the LD-9 `<PackageReference>`-only distribution model end-to-end.
- **FR-3014 (MUST, AC-9.15).** Harness subcommand `run-ac-9.15` (`IncrementalBuild`) composes the canonical consumer csproj, runs `dotnet build -c Debug --nologo /verbosity:detailed` twice back-to-back against the same source tree, captures both stdout+stderr buffers, and asserts: (a) both exit codes 0; (b) first build's stdout contains the substring `Task "GravityDslGenTask"` (the MSBuild canonical detailed-verbosity task-execution marker); (c) second build's stdout does NOT contain that substring (the `Inputs`/`Outputs` short-circuit kicked in). A third sub-step `touch`es the `.gravity` source between builds (updating its mtime) and asserts the next build re-executes the task — i.e., `Task "GravityDslGenTask"` reappears in stdout. Failure modes map to `HARN005` (first build failed), `HARN006` (first build did not run the task), `HARN007` (second build ran the task — incremental short-circuit broken), `HARN008` (touch+rebuild did not re-trigger).
- **FR-3015 (MUST, AC-9.7-pack, LD-22, LD-25).** Harness subcommand `run-ac-9.7-pack` (`PackDeterminism`) runs `dotnet pack Gravity.Dsl.MsBuild.csproj -c Release -o <scratch>/pack-1` and `dotnet pack ... -o <scratch>/pack-2` inside the subcommand's own scratch directory (per LD-25 — no amortised pack across subcommands), normalises both produced `.nupkg` files through the post-pack normaliser (`tools/nupkg-normaliser`), and asserts SHA-256 equality between the two normalised buffers. The same flow repeats for `Gravity.Dsl.Emitter.Sample.Outline.csproj`. The pre-normalisation hashes are also captured and logged so a regression that re-introduces determinism without the normaliser is visible to a reviewer; the assertion is on the **normalised** hashes. Failure surfaces `HARN009` ("pack-determinism harness step failed; non-normalised hashes <h1> <h2>; normalised hashes <h1n> <h2n>").

### 4.3 Post-pack normaliser

- **FR-3020 (MUST, LD-22, "Deterministic output").** A new console project `tools/nupkg-normaliser/Gravity.Dsl.NupkgNormaliser/` accepts `--input <nupkg>` and `--output <nupkg>` arguments, reads the input as a zip stream, and writes a normalised zip with: (a) entries sorted by path under `StringComparer.Ordinal`; (b) entry timestamps zeroed (the minimum-valid ZIP epoch is `1980-01-01T00:00:00Z`, not the Unix epoch — the .NET `ZipArchive` writer rejects pre-1980 timestamps); (c) extra-fields removed (so per-entry `extra` byte sequences from the SDK's zip writer do not leak); (d) the `package/services/metadata/core-properties/<random-guid>.psmdcp` entry renamed to `package/services/metadata/core-properties/<sha256-of-content>.psmdcp` (where the SHA-256 covers the **decompressed** entry bytes, hex-lowercase, the full 64 chars); (e) the **single** `_rels/.rels` `<Relationship>` entry that points at the `.psmdcp` file is updated — its `Target` attribute is rewritten to the renamed `.psmdcp` path AND its `Id` attribute is rewritten to the literal `"R1"` (the implementation spike confirmed both `Target` and `Id` on this row are non-deterministic across packs); no other `<Relationship>` entries are touched. The first `<Relationship>` (which points at the `.nuspec` manifest and carries a content-deterministic `Id` such as `RAC971DF315D82D83`) is **not** rewritten — the spike confirmed it is already byte-stable across packs and rewriting it would introduce divergence the SDK does not produce.
- **FR-3021 (MUST, "Deterministic output", III).** The normaliser is read-only on its input: the input `.nupkg` file is opened with `FileShare.Read`, the output is written atomically (write to `<output>.tmp` then `File.Move` to the final path), and the input is never overwritten in place even when the harness calls with the same path for input and output (in that case the normaliser writes to a sibling temp file and `Move`s with `overwrite: true`). The normaliser never reaches into the source tree, never modifies anything under `<repo>/.git/`, and never spawns child processes.
- **FR-3022 (MUST, IV).** The normaliser CLI surface is additive-only versioned: `--input` and `--output` are stable; future options (`--strip-signatures`, `--canonicalise-nuspec-order`, etc.) are added without removing or renaming existing flags. A `--version` flag prints the normaliser's own semantic version (start at `1.0.0`); the version is bumped by the Phase 9c task list and is independent of the DSL grammar version and the AST version.
- **FR-3023 (MUST, "Deterministic output").** The normaliser is itself byte-deterministic: running `Gravity.Dsl.NupkgNormaliser --input X --output Y` twice against the same input produces byte-identical `Y` outputs. The normaliser is also **idempotent on its own output**: running `NupkgNormalizer.Normalize` three times back-to-back against the same `.nupkg` produces SHA-256-equal results for runs 2 and 3 (idempotence). A unit test under `tools/nupkg-normaliser/Gravity.Dsl.NupkgNormaliser.Tests/` asserts both properties against a fixture `.nupkg` (the fixture is a tiny synthetic `.nupkg` checked into `tests/fixtures/nupkg-normaliser/` so the test does not require a live `dotnet pack` invocation).
- **FR-3024 (MAY).** The normaliser MAY in a future release add a `--verify` mode that takes two `.nupkg` paths and exits 0 iff their normalised forms hash equally. Phase 9c does not deliver `--verify`; the harness implements this comparison itself in `Program.cs`.
